2. 전체 흐름

```
1. 클라이언트가 API Insert 요청

2. Controller -> Service 버퍼 push 만 수행
   - 성공 시 즉시 accepted 응답
   - 버퍼 초과 시 503 반환 (Backpressure)

3. job_queue 반영 타이밍
   - 요청은 계속 버퍼에만 push
   - 0.5초마다 타이머가 한 번 돌 때마다 버퍼에서 최대 200건까지만 꺼내 INSERT (그 틱에 나머지는 버퍼에 남김 → 다음 틱에 이어서 처리)
   - 종료 시에는 버퍼가 빌 때까지 200건 단위로 반복 처리
   전제: DB가 장시간 죽어 있는 상태를 버퍼로 무한히 흡수하지 않음 (네트워크/DB 장애 시 재시도 큐 아님)
   배치 실패 시 같은 묶음을 단건 INSERT로 fallback (데이터 분리용일 뿐, 장애 복구용 재시도 아님)
   단건까지 실패한 건 즉시 flush.log(+ 콘솔)로만 남김 — 재시도 없음
   타이머 틱과 겹치지 않게 flushing 플래그 사용 (버퍼에 남은 건 다음 0.5초 틱에서 처리)
   DB 반영이 멈추면 버퍼만 차고 결국 MAX_BUFFER_SIZE에서 503 → 그건 의도된 백프레셔·트레이드오프

4. Worker가 LISTEN/NOTIFY 신호 수신
   pending 없을 때까지 하나씩 처리
   handle() 실패 시 failed 상태 기록 + handle.log 기록
   LISTEN 커넥션 단절 시 재연결은 코드 예시 참고 (예: 최대 5회, 1초~5초 선형 대기)

5. done 데이터는 주기적으로 삭제
   failed는 남겨두고 수동 확인 후 삭제
```

3. 전체 프로세스 순서도

```
════════════════════════════════════════════════════
  요청 수신
════════════════════════════════════════════════════

클라이언트
  │
  │  POST /api/db-gather
  ▼
GatherController
  │  enqueue(dto)
  │  성공: { status: 'accepted' } / 실패: 503 Service Unavailable
  ▼
GatherService.enqueue()
  │
  ├─▶ 메모리 버퍼 push
  └─▶ MAX_BUFFER_SIZE 초과 시 false 반환


════════════════════════════════════════════════════
  버퍼 -> job_queue (0.5초마다 최대 200건 한 묶음만 INSERT / 처리 중이면 틱 스킵)
════════════════════════════════════════════════════

flushChunk()
  │
  ├─ 버퍼에서 최대 200건 한 번만 추출 (남은 건 다음 틱)
  │
  ├─ createMany() 벌크 INSERT
  │   ├─ 성공 -> 이번 틱 종료
  │   └─ 실패 -> 같은 묶음 단건 INSERT fallback
  │
  └─ 단건도 실패한 건만 flush.log 기록


════════════════════════════════════════════════════
  job_queue -> Worker
════════════════════════════════════════════════════

job_queue INSERT 성공
  │
  ▼
DB 트리거 (statement 단위)
  │  pg_notify('new_job', 'batch')
  ▼
Worker LISTEN 수신
  │
  ▼
pending 소진 루프
  │
  ├─ FOR UPDATE SKIP LOCKED로 1건 획득
  ├─ status -> processing
  ├─ handle(payload)
  │   ├─ 성공 -> done
  │   └─ 실패 -> failed + handle.log
  ▼
pending 없으면 종료


════════════════════════════════════════════════════
  정리
════════════════════════════════════════════════════

cleanup timer
  │
  ├─ done + created_at 24시간 초과 -> DELETE
  └─ failed -> 유지 (수동 확인)
```

4. 왜 이런 구조인가?

| 문제                   | 해결                           |
| -------------------- | ---------------------------- |
| 건당 INSERT DB 왕복 비용 큼 | 버퍼링 + Batch INSERT           |
| 요청 타이밍에 후속 처리가 몰림    | Queue 분리 + Worker 처리         |
| 워커 다중 실행 시 중복 처리     | FOR UPDATE SKIP LOCKED       |
| polling은 불필요 쿼리 많음   | LISTEN/NOTIFY 신호 기반 처리       |
| 벌크 1회 실패로 전체 유실 위험   | 단건 fallback로 성공 가능한 건 반영     |
| 실패 원인 추적 어려움         | flush.log / handle.log 분리 기록 |
| 버퍼 무한 증가 위험          | MAX_BUFFER_SIZE 초과 시 503     |

<br />

### FOR UPDATE SKIP LOCKED — 단일 워커 이상을 위한 설계

```
현재 구조는 단일 워커 기준이지만,
FOR UPDATE SKIP LOCKED를 사용하기 때문에
워커를 수평 확장(멀티 프로세스/멀티 인스턴스)해도
동일 job을 두 워커가 동시에 가져가는 중복 처리가 방지된다.

즉, 지금은 단순하게 단일로 운영하되
나중에 처리량이 부족해지면 워커만 늘려도
DB 레벨에서 중복 없이 안전하게 확장된다.

단, handle()은 멱등성(idempotency) 보장이 필수다.
(done 업데이트 전 프로세스가 종료되면 재기동 시 재처리될 수 있음)
```

5. 트레이드오프 / 리스크 / 크리티컬

<br />

`트레이드오프 - 의도한 것`

```
즉시 accepted 응답
- 클라이언트는 수집 성공으로 인식
- 실제 DB 반영은 flush 시점
- 수집 API 특성상 허용
```

```
flush 단계에서 네트워크/DB 장애를 전제로 한 재시도 없음
- "언젠가 INSERT 되게" 버퍼만 키우는 설계가 아님
- 장애가 길면 버퍼가 한도까지 차고 503으로 거절되는 게 정상 반응 (말이 안 되는 적재를 피함)
- 실패 건은 즉시 로그(flush.log / 콘솔)만 남기고 해당 레코드는 큐까지 못 감 — 운영자가 로그로 조치

정상 운영에서는 flush 단계 에러가 드물다고 두고,
발생하면 장애 대응·DB 복구가 선행 과제지 애플리케이션 레벨 무한 재시도 대상이 아님
```

```
Worker 단건 순차 처리 유지
- 처리량보다 예측 가능성 우선
- 복잡한 동시성 제어 제거
```

<br />

`리스크 - 알고 감수하는 것`

```
메모리 버퍼 유실 가능성
- 비정상 종료 시 버퍼에 남은 데이터 유실 가능
- 종료 훅에서 flush 호출로 완화
```

```
DB/네트워크 장애 시 버퍼 적체
- 재시도로 버퍼만 무한히 받지 않음 → 한도 도달 시 503
- "버퍼 1만 건이 계속 찬다"는 건 운영적으로 DB 살리는 게 먼저인 상태로 두는 선택
```

```
외부 API 장애 시 failed 누적
- 자동 재처리 없음
- 수동 확인 프로세스 필요
```

```
handle() 완료 직후 프로세스 종료 시 중복 처리 가능성
- done 업데이트 전 종료되면 재기동 시 재처리될 수 있음
- handle() 구현은 멱등성(idempotency) 보장이 필요
```

<br />

`크리티컬 - 배포 전 필수 체크리스트`

```
□ flush.log / handle.log 모니터링 체계
  - 로그 파일 확인 책임자 지정
  - 알림 연동(Slack 등) 권장
  - 감지 기준 예시:
    · flush.log에 신규 라인이 발생하면 즉시 알림
    · handle.log 누적 건수가 1시간 내 N건 초과 시 알림
    · 로그 파일 tail -f 또는 filebeat 등으로 실시간 감시 권장

□ 버퍼 임계치 정책 확정
  - MAX_BUFFER_SIZE 기준 운영값 합의 (기본값: 10,000건)
  - 503 반환 시 클라이언트 재시도 정책 합의
    · 예: 지수 백오프(Exponential Backoff) + 최대 재시도 횟수 설정

□ handle() 멱등성 검증
  - 동일 payload로 handle()이 두 번 호출되어도 부작용 없는지 확인
  - 재기동 시 processing → pending 복구 로직 동작 확인 (onApplicationBootstrap)

□ LISTEN 재연결 완전 실패 시 대응
  - MAX_RECONNECT_RETRY 소진 시 process.exit(1)로 프로세스가 종료됨
  - PM2 / systemd 등 프로세스 매니저의 자동 재기동 설정이 반드시 되어 있어야 함
    · PM2 예시: pm2 start dist/main.js --restart-delay=3000
    · 재기동 시 onApplicationBootstrap에서 processing → pending 자동 복구됨
  - 재연결 소진 시점에 Slack 등 알림 발송 연동 권장 (코드 내 TODO 주석 위치 참고)

□ SQL 적용 경로 확정 (아래 8절 참고)
  - setup.sql 단독 vs Prisma migrate + 트리거만 별도 적용 중 하나만 선택
  - 이중 적용 시 인덱스 충돌 발생 가능
```

### 적용 경로 선택 — 반드시 하나만

```
setup.sql 전체 적용 (테이블 + 인덱스 + 트리거)
  → Prisma migrate dev 실행 불필요 (동일 DDL 이중 생성 금지)
  → prisma generate 만 실행하여 클라이언트 타입 생성

Prisma migrate dev 적용 (테이블 + 인덱스)
  → setup.sql의 CREATE TABLE / CREATE INDEX 는 실행하지 않음
  → notify_new_job 트리거 부분만 따로 psql 등으로 실행
  → migrate dev 만으로는 트리거가 생성되지 않으므로 트리거는 반드시 별도 적용

⚠️  두 경로를 동시에 적용하면 인덱스 이름 충돌 또는 중복 인덱스가 생성될 수 있다.
    동일 객체를 두 경로로 이중 생성하지 않는다.
```

<br />

`sql/setup.sql`

```sql
-- job_queue 테이블 (Prisma JobQueue의 @db.Timestamptz(3)와 맞춤)
CREATE TABLE job_queue (
  id           SERIAL PRIMARY KEY,
  type         VARCHAR    NOT NULL,
  payload      JSONB      NOT NULL,
  status       VARCHAR    NOT NULL DEFAULT 'pending',
  error        TEXT,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  processed_at TIMESTAMPTZ
);

CREATE INDEX idx_job_queue_status_created
  ON job_queue (status, created_at);

-- LISTEN/NOTIFY 트리거 (statement 단위)
CREATE OR REPLACE FUNCTION notify_new_job()
RETURNS trigger AS $$
BEGIN
  PERFORM pg_notify('new_job', 'batch');
  RETURN NULL;
END;
$$ LANGUAGE plpgsql;

DROP TRIGGER IF EXISTS trg_notify_new_job ON job_queue;

CREATE TRIGGER trg_notify_new_job
AFTER INSERT ON job_queue
FOR EACH STATEMENT EXECUTE FUNCTION notify_new_job();
```

8. Prisma 설정

<br />

`prisma/schema.prisma`

```prisma
model JobQueue {
  id          Int       @id @default(autoincrement())
  type        String
  payload     Json
  status      String    @default("pending")
  error       String?
  createdAt   DateTime  @default(now()) @map("created_at") @db.Timestamptz(3)
  processedAt DateTime? @map("processed_at") @db.Timestamptz(3)

  @@index([status, createdAt])
  @@map("job_queue")
}
```

※ 이미 있는 `schema.prisma`의 `generator`·`datasource`는 유지하고, **`JobQueue` 모델 블록만 추가**하면 된다.

※ **8절 `setup.sql` 전체를 이미 적용했다면** `migrate dev`를 돌리지 않는다. 이미 DB에 테이블과 인덱스가 존재하기 때문에 충돌이 발생할 수 있다. 이 경우 `prisma generate` 만 실행하면 된다.

※ **`migrate dev`로 테이블을 만들었다면** 8절의 `CREATE TABLE`·인덱스는 실행하지 않고, `notify_new_job` 트리거 블록만 별도로 적용한다.

```bash
npm install pg class-validator class-transformer
npm install -D @types/pg
npx prisma migrate dev   # setup.sql 미적용 시에만 실행
npx prisma generate
```

9. 코드 작성 (심플 구현 예시)

<br />

`요구사항 반영 체크`

```text
1) 0.5초 타이머마다 최대 200건만 DB 반영 (enqueue는 push만). 종료 시 버퍼 빌 때까지 200건 단위 반복
2) 벌크 실패 시 단건 처리, 최종 실패만 flush.log(+ 콘솔) — DB/네트워크 장애 재시도 없음
3) 재시도 없음 (fallback은 불량 건 분리용이지 장애 복구 재시도 아님)
4) 반복문은 간결한 for 문 사용
5) 중첩 try/catch 최소화
6) Promise.race 미사용
7) 비즈니스용 플래그/재시도 최소화 (LISTEN 재연결 등 운영 최소 가드는 예외 가능)
```

`gather/gather.controller.ts`

```ts
import {
  Body,
  Controller,
  HttpCode,
  Post,
  ServiceUnavailableException,
} from '@nestjs/common';
import { DbGatherDto } from './dto/db-gather.dto';
import { GatherService } from './gather.service';

@Controller('api')
export class GatherController {
  constructor(private readonly gatherService: GatherService) {}

  @Post('db-gather')
  @HttpCode(202)
  receive(@Body() dto: DbGatherDto) {
    const accepted = this.gatherService.enqueue(dto);
    if (!accepted) {
      throw new ServiceUnavailableException('buffer is full, try again later');
    }
    return { status: 'accepted' };
  }
}
```

`gather/gather.service.ts`

```ts
import {
  Injectable,
  OnApplicationBootstrap,
  OnApplicationShutdown,
} from '@nestjs/common';
import { PrismaService } from '../prisma.service';
import { DbGatherDto } from './dto/db-gather.dto';
import { appendJsonLog } from '../common/append-json-log';

interface BufferItem {
  dto: DbGatherDto;
}

@Injectable()
export class GatherService implements OnApplicationBootstrap, OnApplicationShutdown {
  private readonly buffer: BufferItem[] = [];
  private timer?: NodeJS.Timeout;
  /** 현재 틱의 flush가 진행 중인지 여부. 타이머 틱 중복 실행 방지용 */
  private flushing = false;
  /** 진행 중인 flushChunk()가 완료되면 resolve되는 Promise. 종료 시 대기에 사용 */
  private flushingDone: Promise<void> = Promise.resolve();

  private readonly FLUSH_INTERVAL_MS = 500;
  /** 한 틱(0.5초)당 최대 DB로 넘기는 건수 */
  private readonly BATCH_SIZE = 200;
  private readonly MAX_BUFFER_SIZE = 10000;

  constructor(private readonly prisma: PrismaService) {}

  onApplicationBootstrap() {
    this.timer = setInterval(() => void this.flushChunk(), this.FLUSH_INTERVAL_MS);
  }

  async onApplicationShutdown() {
    if (this.timer) clearInterval(this.timer);

    // 진행 중인 flush가 완료될 때까지 Promise로 대기 (폴링 방식 대신 완료 신호 기반)
    await this.flushingDone;

    while (this.buffer.length > 0) {
      await this.flushChunk();
    }
  }

  /** 버퍼에만 적재. DB 반영은 타이머·종료 시에만 */
  enqueue(dto: DbGatherDto): boolean {
    if (this.buffer.length >= this.MAX_BUFFER_SIZE) return false;
    this.buffer.push({ dto });
    return true;
  }

  /** 타이머(0.5초)·종료 시 동일: 버퍼에서 최대 200건 한 묶음만 INSERT */
  private async flushChunk(): Promise<void> {
    if (this.flushing || this.buffer.length === 0) return;

    this.flushing = true;

    // 완료 시점을 외부에서 await할 수 있도록 Promise를 멤버에 보관
    let resolveFlushing!: () => void;
    this.flushingDone = new Promise<void>((resolve) => {
      resolveFlushing = resolve;
    });

    try {
      const items = this.buffer.splice(0, this.BATCH_SIZE);

      try {
        await this.prisma.jobQueue.createMany({
          data: items.map(({ dto }) => ({
            type: 'DB_GATHER',
            payload: dto,
            status: 'pending',
          })),
        });
      } catch (bulkError) {
        for (const item of items) {
          try {
            await this.prisma.jobQueue.create({
              data: {
                type: 'DB_GATHER',
                payload: item.dto,
                status: 'pending',
              },
            });
          } catch (singleError) {
            await this.safeWriteFlushLog({
              context: 'GatherService',
              dto: item.dto,
              bulkError: String(bulkError),
              singleError: String(singleError),
              timestamp: new Date().toISOString(),
            });
          }
        }
      }
    } catch (e) {
      console.error('[flush failed]', e);
    } finally {
      this.flushing = false;
      resolveFlushing(); // 대기 중인 종료 훅에 완료 신호 전달
    }
  }

  private async safeWriteFlushLog(payload: Record<string, unknown>) {
    try {
      await appendJsonLog('flush.log', payload);
    } catch (e) {
      console.error('[flush.log write failed]', e, payload);
    }
  }
}
```

> **변경 포인트 — `onApplicationShutdown` 대기 방식**
>
> 기존: `while (this.flushing) { await new Promise(r => setImmediate(r)); }` — setImmediate를 빠르게 반복 폴링하는 구조로, 완료 시점을 직접 알 수 없음.
>
> 변경: `flushChunk` 시작 시 내부적으로 Promise를 생성하고 `finally`에서 resolve. 종료 훅은 해당 Promise를 `await`하여 완료 신호를 받는 즉시 다음 단계로 진행. 불필요한 tick 소비 없이 의도가 코드에서 명확하게 읽힘.

```ts
import {
  Injectable,
  OnApplicationBootstrap,
  OnApplicationShutdown,
} from '@nestjs/common';
import { PrismaService } from '../prisma.service';
import { Client } from 'pg';
import { appendJsonLog } from '../common/append-json-log';

/** handle()에 전달되는 payload 구조. 실제 비즈니스 스펙에 맞게 확장한다. */
interface GatherPayload {
  dbCode: string;
  contractNo: string;
  payload: Record<string, unknown>;
}

@Injectable()
export class GatherWorker implements OnApplicationBootstrap, OnApplicationShutdown {
  private client?: Client;
  private cleanupTimer?: NodeJS.Timeout;
  private vacuumTimer?: NodeJS.Timeout;
  private reindexTimer?: NodeJS.Timeout;
  private reconnectTimer?: NodeJS.Timeout;
  private readonly CLEANUP_INTERVAL = 60 * 60 * 1000;
  private readonly RETENTION_MS = 24 * 60 * 60 * 1000;
  private readonly MAX_RECONNECT_RETRY = 5;
  private isReconnecting = false;
  private isShuttingDown = false;

  constructor(private readonly prisma: PrismaService) {}

  async onApplicationBootstrap() {
    await this.prisma.jobQueue.updateMany({
      where: { status: 'processing' },
      data: { status: 'pending' },
    });
    await this.connectListen();
    await this.processPending();
    this.cleanupTimer = setInterval(() => void this.cleanupDone(), this.CLEANUP_INTERVAL);

    // Dead Tuple 정리 + 통계 갱신: 매일 새벽 4시
    // 대량 DELETE가 반복되는 테이블은 autovacuum이 따라가지 못할 수 있다.
    this.vacuumTimer = this.scheduleDailyAt(4, 0, () => void this.runVacuum());

    // 인덱스 비대화(Index Bloat) 방지: 매주 일요일 새벽 3시
    // REINDEX CONCURRENTLY — 잠금 없이 백그라운드 재구성 (PostgreSQL 12+)
    // REINDEX INDEX (잠금 있음, 운영 중 사용 금지)는 사용하지 않는다.
    this.reindexTimer = this.scheduleWeeklyAt(0, 3, 0, () => void this.runReindex());
  }

  async onApplicationShutdown() {
    this.isShuttingDown = true;
    if (this.cleanupTimer) clearInterval(this.cleanupTimer);
    if (this.vacuumTimer) clearTimeout(this.vacuumTimer);
    if (this.reindexTimer) clearTimeout(this.reindexTimer);
    if (this.reconnectTimer) clearTimeout(this.reconnectTimer);
    if (!this.client) return;
    this.client.removeAllListeners();
    await this.client.end();
  }

  private async connectListen() {
    this.client = new Client({ connectionString: process.env.DATABASE_URL });
    await this.client.connect();
    await this.client.query('LISTEN new_job');
    this.client.on('notification', () => void this.processPending());
    this.client.on('error', (err) => {
      // 이미 끊긴 커넥션의 에러는 예상 범위이므로 warn 수준으로 기록
      console.warn('[LISTEN client error]', err?.message ?? err);
      void this.reconnect();
    });
    this.client.on('end', () => {
      console.warn('[LISTEN client ended]');
      void this.reconnect();
    });
  }

  private async reconnect() {
    // 같은 시점에 error/end가 연속 발생해도 재연결은 1회만 수행
    // 종료 중이면 재연결을 시도하지 않음
    if (this.isReconnecting || this.isShuttingDown) return;
    this.isReconnecting = true;

    try {
      if (this.client) {
        this.client.removeAllListeners();
        // 이미 끊긴 커넥션일 수 있으므로 실패해도 재연결 흐름은 계속 진행
        await this.client.end().catch((e) =>
          console.warn('[reconnect] client.end() ignored:', e?.message ?? e),
        );
      }
    } catch {}

    let reconnected = false;
    for (let retry = 1; retry <= this.MAX_RECONNECT_RETRY; retry++) {
      if (this.isShuttingDown) break;

      const delayMs = retry * 1000; // 1초, 2초, 3초, 4초, 5초
      await new Promise<void>((resolve) => {
        this.reconnectTimer = setTimeout(resolve, delayMs);
      });

      try {
        await this.connectListen();
        await this.processPending();
        reconnected = true;
        break;
      } catch (e) {
        console.error(`[reconnect failed] retry=${retry}/${this.MAX_RECONNECT_RETRY}`, e);
      }
    }

    // 재연결 전부 실패 시: 워커가 조용히 죽는 것을 방지하기 위해 명시적으로 기록하고 프로세스를 종료한다.
    // 프로세스 매니저(PM2, systemd 등)가 자동으로 재기동하도록 위임하는 것이 단순하고 안전하다.
    // 재기동 시 onApplicationBootstrap에서 processing → pending 복구가 자동으로 실행된다.
    if (!reconnected && !this.isShuttingDown) {
      console.error(
        `[reconnect exhausted] LISTEN 재연결 ${this.MAX_RECONNECT_RETRY}회 전부 실패. 프로세스를 종료합니다.`,
      );
      // 알림 연동이 있다면 이 시점에 Slack 등으로 알림 발송 권장
      process.exit(1);
    }

    this.isReconnecting = false;
  }

  private async processPending() {
    // 1. 종료 중이면 실행 방지
    if (this.isShuttingDown) return;

    while (true) {
      // 2. 이벤트 루프에 제어권을 양보.
      //    job 획득 직전에 위치시켜, pending이 없어 break되는 경우에는 양보하지 않음
      await new Promise((resolve) => setImmediate(resolve));

      // 3. 종료 신호가 감지되면 루프 즉시 탈출
      if (this.isShuttingDown) break;

      try {
        // 4. 트랜잭션 범위 시작
        const job = await this.prisma.$transaction(async (tx) => {
          const rows = await tx.$queryRaw<{ id: number; payload: unknown }[]>`
            SELECT id, payload
            FROM job_queue
            WHERE status = 'pending'
            ORDER BY created_at ASC
            LIMIT 1
            FOR UPDATE SKIP LOCKED
          `;

          if (rows.length === 0) return null;

          await tx.jobQueue.update({
            where: { id: rows[0].id },
            data: { status: 'processing' },
          });

          return rows[0];
        }, { timeout: 10000 }); // 트랜잭션 타임아웃 10초

        // 5. 처리할 작업이 없으면 루프 종료 (다음 LISTEN 신호를 기다림)
        if (!job) break;

        // 6. 실제 비즈니스 로직 실행 (handle)
        try {
          await this.handle(job.payload as GatherPayload);

          // 성공 시 완료 처리
          await this.prisma.jobQueue.update({
            where: { id: job.id },
            data: { status: 'done', processedAt: new Date() },
          });
        } catch (e) {
          // 비즈니스 로직 실패 시 failed 처리 및 로그 기록
          const error = e instanceof Error ? e.message : String(e);
          await this.prisma.jobQueue.update({
            where: { id: job.id },
            data: { status: 'failed', error, processedAt: new Date() },
          });
          await this.safeWriteHandleLog({
            context: 'GatherWorker',
            jobId: job.id,
            payload: job.payload,
            error,
            timestamp: new Date().toISOString(),
          });
        }
      } catch (transactionError) {
        // 7. DB 연결 끊김 등 트랜잭션 자체 실패 시 예외 처리
        console.error('[Worker Transaction Error]', transactionError);

        // DB가 일시적으로 아픈 상태일 수 있으므로 1초 쉬었다가 루프 탈출
        // (탈출하면 나중에 LISTEN 재연결 로직이 다시 살려냅니다)
        await new Promise((resolve) => setTimeout(resolve, 1000));
        break;
      }
    }
  }

  /**
   * 실제 비즈니스 처리 로직.
   *
   * ⚠️  멱등성(idempotency) 보장 필수:
   *   done 업데이트 전에 프로세스가 종료되면 재기동 시 동일 job이 재처리될 수 있다.
   *   동일 payload로 두 번 호출되어도 부작용이 없도록 구현해야 한다.
   *   예: 외부 API 호출 전 중복 여부 확인, upsert 사용, 고유 키 기반 방어 등.
   *
   * ⚠️  외부 API 호출 시 반드시 타임아웃을 걸 것:
   *   handle()이 hang되면 워커 전체가 멈춘다.
   *   Promise.race는 타임아웃 후에도 원래 Promise가 백그라운드에서 계속 실행되므로 사용하지 않는다.
   *   AbortController로 요청 자체를 취소하는 방식을 권장한다. (아래 예시 참고)

  private async cleanupDone() {
    const cutoff = new Date(Date.now() - this.RETENTION_MS);
    await this.prisma.jobQueue.deleteMany({
      where: {
        status: 'done',
        createdAt: { lt: cutoff },
      },
    });
  }

  /**
   * Dead Tuple 정리 + 쿼리 플래너 통계 갱신.
   *
   * 대량 INSERT/DELETE가 반복되는 테이블은 autovacuum이 따라가지 못해
   * Dead Tuple이 누적되고 쿼리 성능이 저하될 수 있다.
   * VACUUM ANALYZE를 주기적으로 실행해 Dead Tuple을 정리하고
   * 실행 계획에 쓰이는 통계를 최신 상태로 유지한다.
   *
   * 실행 중 테이블 잠금 없음 — 운영 중 안전하게 실행 가능.
   */
  private async runVacuum() {
    try {
      console.log('[maintenance] VACUUM ANALYZE job_queue 시작');
      // Prisma는 VACUUM을 지원하지 않으므로 $executeRawUnsafe 사용
      await this.prisma.$executeRawUnsafe('VACUUM ANALYZE job_queue');
      console.log('[maintenance] VACUUM ANALYZE job_queue 완료');
    } catch (e) {
      console.error('[maintenance] VACUUM ANALYZE 실패', e);
    } finally {
      // 다음 날 새벽 4시에 다시 실행되도록 재예약
      if (!this.isShuttingDown) {
        this.vacuumTimer = this.scheduleDailyAt(4, 0, () => void this.runVacuum());
      }
    }
  }

  /**
   * 인덱스 비대화(Index Bloat) 방지를 위한 REINDEX.
   *
   * pending → done → DELETE 사이클이 반복되면 인덱스 내부에
   * 사용되지 않는 페이지가 쌓여 스캔 성능이 저하될 수 있다.
   *
   * REINDEX INDEX CONCURRENTLY — PostgreSQL 12+
   *   잠금 없이 백그라운드에서 인덱스를 재구성한다.
   *   운영 중 INSERT/SELECT에 영향을 주지 않는다.
   *
   * ⚠️  REINDEX INDEX (CONCURRENTLY 없이)는 ShareLock을 걸어
   *     운영 중 INSERT/SELECT를 블로킹한다 — 절대 사용하지 않는다.
   *
   * ⚠️  REINDEX CONCURRENTLY는 트랜잭션 블록 안에서 실행할 수 없다.
   *     Prisma $transaction이 아닌 $executeRawUnsafe로 직접 실행한다.
   */
  private async runReindex() {
    try {
      console.log('[maintenance] REINDEX INDEX CONCURRENTLY 시작');
      await this.prisma.$executeRawUnsafe(
        'REINDEX INDEX CONCURRENTLY idx_job_queue_status_created',
      );
      console.log('[maintenance] REINDEX INDEX CONCURRENTLY 완료');
    } catch (e) {
      console.error('[maintenance] REINDEX 실패', e);
    } finally {
      // 다음 주 일요일 새벽 3시에 다시 실행되도록 재예약
      if (!this.isShuttingDown) {
        this.reindexTimer = this.scheduleWeeklyAt(0, 3, 0, () => void this.runReindex());
      }
    }
  }

  /**
   * 매일 특정 시각(hour:minute)에 콜백을 한 번 실행하는 타이머를 예약한다.
   * setInterval 대신 setTimeout 체인을 사용해 실행 시각의 정확도를 높인다.
   */
  private scheduleDailyAt(hour: number, minute: number, cb: () => void): NodeJS.Timeout {
    const now = new Date();
    const next = new Date(now);
    next.setHours(hour, minute, 0, 0);
    if (next <= now) next.setDate(next.getDate() + 1); // 오늘 이미 지났으면 내일로
    return setTimeout(cb, next.getTime() - now.getTime());
  }

  /**
   * 매주 특정 요일(dayOfWeek 0=일)·시각(hour:minute)에 콜백을 한 번 실행하는 타이머를 예약한다.
   */
  private scheduleWeeklyAt(
    dayOfWeek: number,
    hour: number,
    minute: number,
    cb: () => void,
  ): NodeJS.Timeout {
    const now = new Date();
    const next = new Date(now);
    next.setHours(hour, minute, 0, 0);
    const daysUntil = (dayOfWeek - now.getDay() + 7) % 7 || 7; // 오늘이면 다음 주로
    next.setDate(next.getDate() + daysUntil);
    return setTimeout(cb, next.getTime() - now.getTime());
  }

  private async safeWriteHandleLog(payload: Record<string, unknown>) {
    try {
      await appendJsonLog('handle.log', payload);
    } catch (e) {
      console.error('[handle.log write failed]', e, payload);
    }
  }
}

> **변경 포인트 요약**
>
> `connectListen` — `error` / `end` 이벤트 핸들러에 `console.warn` 추가. 이미 끊긴 커넥션의 에러는 예상 범위이므로 error가 아닌 warn 수준으로 기록하여 운영 시 노이즈를 낮춘다.
>
> `reconnect` — `client.end()` 실패를 `.catch()`로 처리하고 warn 기록. 빈 catch 블록에서 묵살하던 것을 이유 있는 무시로 변경. 5회 전부 실패 시 `process.exit(1)`로 명시적 종료 추가 — 워커가 조용히 죽어 아무도 모르는 상태를 방지하고 PM2/systemd 재기동에 위임한다.
>
> `processPending` — `setImmediate` 위치를 루프 최상단에서 job 획득 직전으로 이동. pending이 없어 즉시 break되는 경우 불필요한 tick 소비를 방지.
>
> `handle` — `void payload` 제거. 파라미터명을 `_payload`로 변경(미사용 파라미터 관례)하고 `GatherPayload` 타입 명시. JSDoc으로 멱등성 보장 요건 및 타임아웃 구현 가이드 추가. `AbortController` 기반 타임아웃 예시 코드 포함 — `Promise.race`와 달리 타임아웃 시 요청 자체가 취소되어 백그라운드 실행 문제가 없다.
>
> `runVacuum` / `runReindex` / `scheduleDailyAt` / `scheduleWeeklyAt` 추가 — DB 유지보수를 애플리케이션 레벨에서 자동화한다. `VACUUM ANALYZE`는 매일 새벽 4시에 Dead Tuple을 정리하고 쿼리 플래너 통계를 갱신한다. `REINDEX INDEX CONCURRENTLY`는 매주 일요일 새벽 3시에 잠금 없이 인덱스를 재구성한다. `setInterval` 대신 `setTimeout` 체인(`scheduleDailyAt` / `scheduleWeeklyAt`)을 사용해 특정 시각에 정확히 실행되도록 한다. Prisma는 `VACUUM` / `REINDEX CONCURRENTLY`를 트랜잭션으로 감싸지 않아야 하므로 `$executeRawUnsafe`로 직접 실행한다.

11. polling vs LISTEN/NOTIFY 비교

| 항목     | polling     | LISTEN/NOTIFY     |
| ------ | ----------- | ----------------- |
| DB 쿼리  | 주기적으로 고정 발생 | 이벤트 시점에만 발생       |
| 반응 속도  | 주기 지연 발생    | 거의 즉시 반응          |
| 유휴 리소스 | 불필요 소모      | 상대적으로 절약          |
| 구현 난이도 | 단순          | 초기 설정 + 재연결 처리 필요 |

12. job 정리 전략 및 DB 유지보수

| 대상 | 방식 | 주기 |
| ------ | --------------------- | --- |
| done 레코드 | 자동 삭제 (24시간 보존) | 1시간마다 |
| failed 레코드 | 수동 확인 후 삭제 | — |
| Dead Tuple | `VACUUM ANALYZE job_queue` | 매일 새벽 4시 |
| Index Bloat | `REINDEX INDEX CONCURRENTLY` | 매주 일요일 새벽 3시 |

```
Dead Tuple (VACUUM ANALYZE)
  - INSERT/DELETE가 반복되면 PostgreSQL MVCC 구조상 삭제된 행이
    즉시 제거되지 않고 Dead Tuple로 남는다.
  - autovacuum이 기본 처리하지만 대량 트래픽 시 따라가지 못할 수 있다.
  - VACUUM ANALYZE는 Dead Tuple 정리 + 쿼리 플래너 통계 갱신을 동시에 수행한다.
  - 실행 중 테이블 잠금 없음 — 운영 중 안전.

Index Bloat (REINDEX CONCURRENTLY)
  - pending → done → DELETE 사이클이 반복되면 인덱스 내부에
    사용되지 않는 페이지(Bloat)가 쌓여 인덱스 스캔 성능이 저하된다.
  - REINDEX INDEX CONCURRENTLY는 잠금 없이 백그라운드에서 재구성한다. (PostgreSQL 12+)
  - REINDEX INDEX (CONCURRENTLY 없음)는 ShareLock 발생 — 절대 사용하지 않는다.
  - 두 작업 모두 GatherWorker 내 스케줄러가 자동 실행하므로 수동 개입 불필요.
    (로그: [maintenance] VACUUM ANALYZE / REINDEX INDEX CONCURRENTLY)
```

로그 종류:

app.log (선택)
- Nest 기본 로거 등으로 서버 공통 로그 운영 시 사용 (예시 코드에는 파일 출력 없음)

flush.log
- 벌크 INSERT 실패 후 단건 INSERT도 실패한 건
- 입력 데이터 품질/DB 오류 추적용

handle.log
- Worker handle() 실패 건
- 외부 연동/후속 처리 오류 추적용
```
