# 대량 INSERT를 안정적으로 처리하기 위한 구조 갖추기

<br />
<br />

* 적은 리소르를 사용해서 대량의 Insert를 처리하는 방법을 알아보자

---

```
개발자는 항상 트레이드오프를 해야 한다.

완전하고 충분한 환경은 없기에,
항상 최적화를 고려하고 적은 리소르를 최대로 사용해야 한다.

싱글 스레드 환경에서 가장 적은 리소스로 대용량 Insert 처리해보자.
```

1. 핵심은 버퍼링, 배치, 큐이다.

```
아주 낮은 사양에 인스턴스가 있다고 가정해보자.

대량의 Insert를 처리할 수 있을까?

방법은 최대한 모으고 (버퍼링),
과정을 생략하고,
조금씩 가져와서 처리하는 것이다. (배치 + 큐)

여기서는 인스턴스의 리소르를 최대한으로 활용하기 위해
가벼운 노드와 백엔드 프레임워크인 NestJs를 사용한다.
```

2. 시나리오

```
클라이언트가 네이버에 메인 지면인
스페셜 DA를 구매했다.

랜딩페이지를 만들었지만
서버는 준비가 되지 않은 상황이다.

기존의 단일 인스턴스 (물리 서버일 수도 있고 클라우드일 수도 있는) 는
업 스케일도 불가능한 상황이다.
```

<br />
<br />

3. 전체 흐름

```
1. 이벤트 랜딩에서 DB Gathering (클라이언트는 API Insert 요청)

2. Controller -> Service 버퍼에 push (즉시 응답 -> 트레이드오프)

3. 버퍼 200건 or 1초마다 -> Batch INSERT -> job_queue 테이블

4. Worker가 2초마다 polling -> pending job 하나씩 처리
```

<br />
<br />
<br />

4. 왜 이런 구조인가?

| 문제 | 해결 |
|------|------|
| 건당 INSERT -> DB 왕복 비용 | 버퍼링 + Batch INSERT |
| 후속 처리 부하가 요청 타임에 집중 | Queue로 분리 -> Worker가 순서대로 처리 |
| Worker 여러 대 중복 처리 | FOR UPDATE SKIP LOCKED |
| 서버 재시작 시 버퍼 데이터 유실 | onApplicationShutdown에서 flush |

`gather.service.ts`

```ts
import { Injectable, Logger, OnApplicationBootstrap, OnApplicationShutdown } from '@nestjs/common';
import { DataSource } from 'typeorm';
import { DbGatherDto } from '../dto/db-gather.dto';
import { JobQueue } from '../entities/job-queue.entity';

/**
 * 버퍼링 + Batch Insert 서비스
 *
 * 왜 버퍼링을 하냐?
 * - 건당 INSERT는 매번 DB 왕복 -> 트래픽 몰리면 커넥션 풀 고갈
 * - 버퍼에 모았다가 한번에 INSERT -> DB 부하 대폭 감소
 *
 * flush 조건 (둘 중 하나라도 충족되면)
 * 1. 버퍼가 BUFFER_LIMIT 이상 쌓였을 때 (즉시)
 * 2. FLUSH_INTERVAL 시간이 지났을 때 (타이머)
 */
@Injectable()
export class GatherService implements OnApplicationBootstrap, OnApplicationShutdown {
  private readonly logger = new Logger(GatherService.name);

  // 메모리 버퍼 (싱글톤이라 앱 내내 같은 인스턴스 -> 요청 간 데이터 유지됨)
  private buffer: DbGatherDto[] = [];
  private timer: NodeJS.Timeout;

  private readonly FLUSH_INTERVAL = 500; // 0.5초마다 자동 flush
  private readonly BUFFER_LIMIT = 200;   // 200건 쌓이면 즉시 flush

  constructor(private readonly dataSource: DataSource) {}

  /**
   * 앱이 완전히 뜬 후 한 번만 호출됨 (NestJS 라이프사이클 훅)
   * constructor에서 시작하면 모듈이 덜 뜬 상태일 수 있어서 여기서 시작
   */
  onApplicationBootstrap() {
    // 타이머는 요청과 무관하게 딱 한 번만 시작
    // 요청마다 시작하면 타이머가 중복 생성됨
    this.timer = setInterval(() => this.flush(), this.FLUSH_INTERVAL);
    this.logger.log('버퍼 타이머 시작');
  }

  /**
   * 앱 종료 직전 호출됨 (NestJS 라이프사이클 훅)
   * main.ts에서 app.enableShutdownHooks() 필수
   * 없으면 배포/재시작 시 버퍼에 남은 데이터 유실
   */
  onApplicationShutdown() {
    clearInterval(this.timer);
    this.flush(); // 남은 버퍼 마저 처리
    this.logger.log('버퍼 타이머 종료');
  }

  /**
   * Controller에서 호출 (await 안 함)
   * 버퍼에 넣기만 하고 즉시 리턴 -> Controller가 바로 응답 가능
   */
  enqueue(dto: DbGatherDto) {
    this.buffer.push(dto);

    // 200건 넘으면 타이머 기다리지 않고 즉시 flush
    if (this.buffer.length >= this.BUFFER_LIMIT) {
      this.flush();
    }
  }

  private async flush() {
    if (this.buffer.length === 0) return;

    // 현재 버퍼 스냅샷 뜨고 비우기
    // splice로 한번에 처리해야 flush 도중 들어온 데이터 안 잃음
    const jobs = this.buffer.splice(0, this.buffer.length);

    try {
      // repo.save(배열)은 내부적으로 건당 처리할 수 있음
      // createQueryBuilder insert -> 진짜 SQL 한방 (bulk insert)
      // INSERT INTO job_queue (type, payload, status) VALUES ($1,$2,$3), ($4,$5,$6) ...
      await this.dataSource
        .createQueryBuilder()
        .insert()
        .into(JobQueue)
        .values(
          jobs.map((dto) => ({
            type: 'DB_GATHER',
            payload: dto,
            status: 'pending',
          })),
        )
        .execute();

      this.logger.log(`Flush 완료: ${jobs.length}건`);
    } catch (e) {
      this.logger.error(`Flush 실패: ${e.message}`);
      // 실패한 데이터 버퍼 앞에 다시 넣어서 다음 flush 때 재시도
      this.buffer.unshift(...jobs);
    }
  }
}
```

`gather.worker.ts`

```ts
import { Injectable, Logger, OnApplicationBootstrap } from '@nestjs/common';
import { DataSource } from 'typeorm';
import { JobQueue } from '../entities/job-queue.entity';

/**
 * 큐 Worker
 *
 * 왜 큐를 쓰냐?
 * - INSERT 부하와 후속 처리 부하를 분리하기 위함
 * - 트래픽이 몰려도 INSERT만 빠르게 받고, 처리는 순서대로 여유있게
 * - 후속 처리가 단순 INSERT 하나면 큐 효과 미미
 *   but 가공 로직, 외부 API 연동 등 무거운 작업이면 큐가 빛남
 *
 * polling 방식
 * - 1초마다 pending 상태 job을 하나씩 꺼내서 처리
 * - FOR UPDATE SKIP LOCKED로 Worker가 여러 대여도 중복 처리 방지
 */
@Injectable()
export class GatherWorker implements OnApplicationBootstrap {
  private readonly logger = new Logger(GatherWorker.name);

  constructor(private readonly dataSource: DataSource) {}

  // 앱 시작 시 polling 루프 시작
  onApplicationBootstrap() {
    this.poll();
  }

  private async poll() {
    // 서버 살아있는 동안 계속 반복
    while (true) {
      await this.processNext();
      await this.sleep(1000); // 1초 대기
    }
  }

  private async processNext() {
    await this.dataSource.transaction(async (manager) => {
      // FOR UPDATE SKIP LOCKED
      // - FOR UPDATE: 가져온 row에 락 걸기
      // - SKIP LOCKED: 이미 락 걸린 row는 건너뜀
      // -> Worker 여러 대 돌아도 같은 job 중복 처리 안 됨
      const job = await manager
        .createQueryBuilder(JobQueue, 'job')
        .where('job.status = :status', { status: 'pending' })
        .orderBy('job.createdAt', 'ASC') // FIFO (먼저 들어온 것부터)
        .limit(1)
        .setLock('pessimistic_write_or_fail')
        .getOne();

      // pending job 없으면 그냥 리턴
      if (!job) return;

      // processing으로 변경 (다른 Worker가 못 가져가게)
      job.status = 'processing';
      await manager.save(job);

      try {
        await this.handle(job.payload);
        job.status = 'done';
        job.processedAt = new Date();
      } catch (e) {
        // 실패해도 status 기록해두면 나중에 재처리 가능
        job.status = 'failed';
        job.error = e.message;
        this.logger.error(`Job ${job.id} 실패: ${e.message}`);
      }

      await manager.save(job);
    });
  }

  /**
   * 실제 비즈니스 로직
   * - 보험 데이터 가공
   * - 실제 테이블에 저장
   * - 외부 API 연동 등
   */
  private async handle(payload: any) {
    this.logger.log(`처리중: ${JSON.stringify(payload)}`);
    // TODO: 실제 처리 로직
  }

  private sleep(ms: number) {
    return new Promise((resolve) => setTimeout(resolve, ms));
  }
}
```

`job-queue.entity.ts`

```ts
import {
  Column,
  CreateDateColumn,
  Entity,
  PrimaryGeneratedColumn,
} from 'typeorm';

/**
 * 큐 테이블
 * - 특별한 테이블 타입이 아닌 일반 테이블
 * - INSERT 후 Worker가 순서대로 꺼내서 처리하는 용도
 */
@Entity('job_queue')
export class JobQueue {
  @PrimaryGeneratedColumn()
  id: number;

  // 작업 종류 (여러 타입의 작업을 한 테이블에서 관리 가능)
  @Column()
  type: string;

  // 보험사에서 온 raw 데이터 (구조가 유동적이라 jsonb로)
  @Column({ type: 'jsonb' })
  payload: object;

  // pending   : 처리 대기
  // processing: Worker가 가져가서 처리중
  // done      : 처리 완료
  // failed    : 처리 실패
  @Column({ default: 'pending' })
  status: string;

  // 실패 시 에러 메시지 저장
  @Column({ nullable: true })
  error: string;

  @CreateDateColumn()
  createdAt: Date;

  // Worker 처리 완료 시점
  @Column({ nullable: true })
  processedAt: Date;
}
```

`SQL`

```sql
-- job_queue 테이블
-- 일반 테이블과 동일, 용도와 조회 패턴이 다를 뿐
CREATE TABLE job_queue (
  id           SERIAL PRIMARY KEY,
  type         VARCHAR   NOT NULL,
  payload      JSONB     NOT NULL,
  status       VARCHAR   NOT NULL DEFAULT 'pending',
  error        TEXT,
  created_at   TIMESTAMP NOT NULL DEFAULT NOW(),
  processed_at TIMESTAMP
);

-- Worker polling용 인덱스
-- status + created_at 복합 인덱스 -> pending 조회 + FIFO 정렬 최적화
-- INSERT 성능에 거의 영향 없음 (이것만 있으면 충분)
CREATE INDEX idx_job_queue_status_created
  ON job_queue (status, created_at);
```
