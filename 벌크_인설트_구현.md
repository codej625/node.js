1. 시나리오

```
낮은 사양의 단일 인스턴스 서버.

클라이언트 이벤트로 단기간에 트래픽이 집중된다.

스케일 업 / 스케일 아웃이 당장 어렵다.

[해결 전략]
· 요청을 메모리 버퍼에 모은다 (버퍼링)
· 일정 주기로 DB에 묶어서 반영한다 (배치 INSERT)
· 후속 처리는 별도 Job Queue가 순차적으로 담당한다

* 사용 스택 NestJS + Prisma + PostgreSQL
```

2. 전체 흐름

```
[클라이언트]
· POST /api/db-gather

▼
```

```
[QueueController]
· 버퍼에 push만 수행
· 성공 -> 202 accepted
· 버퍼 초과 -> 503 (Backpressure)

▼
```

```
[QueueBuffer (메모리)]
· 0.5초마다 최대 200건을 꺼내 job_queue 테이블에 INSERT
· 틱(tick) 처리 중이면 다음 틱으로 스킵
· 벌크 INSERT 실패 시 단건 fallback
· 단건도 실패 시 flush.log 기록 (재시도 없음)
· 종료 시 버퍼가 빌 때까지 200건 단위 반복 처리

▼
```

```
[job_queue 테이블]
· INSERT 성공 시 DB 트리거가 pg_notify('new_job') 발행

▼
```

```
[JobProcessor (LISTEN 기반)]
· LISTEN new_job 신호 수신
· pending 소진 루프: FOR UPDATE SKIP LOCKED 로 1건씩 획득
· pending 없음 → 루프 탈출 후 다음 LISTEN 신호 대기
· Safety Polling: SAFETY_POLLING_INTERVAL_MS 주기마다 processPending() 강제 실행 (기본값 5분, 신호 누락 대비)
· handle() 성공 -> done
· handle() 실패 -> failed + handle.log 기록

▼
```

```
[정리 스케줄러]
· done 레코드: 24시간 보존 후 자동 삭제 (1시간 주기)
· failed 레코드: 수동 확인 후 삭제
· VACUUM ANALYZE: 매일 새벽 4시
· REINDEX CONCURRENTLY: 매주 일요일 새벽 3시

END
```

<br />

`Backpressure(배압) 의도`

```
· DB 장애가 길어지면 버퍼가 점점 차고, MAX_BUFFER_SIZE 도달 시 503을 반환
· "언젠가 반드시 DB에 적재"하려고 버퍼를 무한히 키우는 구조가 아님
· 장애 시에는 DB 복구가 선행이며, 애플리케이션 레벨의 무한 재시도는 의도적으로 배제
```

3. 왜 이런 구조인가?

| 문제 | 해결 |
| --- | --- |
| 건당 INSERT -> DB 왕복 비용 과다 | 버퍼링 + 배치 INSERT |
| 후속 처리가 요청 타이밍에 집중 | Job Queue 분리 + 순차 처리 |
| 다중 프로세스 동시 처리 시 중복 | FOR UPDATE SKIP LOCKED |
| polling -> 불필요한 쿼리 발생 | LISTEN/NOTIFY 신호 기반 |
| 벌크 실패 시 전체 유실 위험 | 단건 fallback으로 분리 반영 |
| 실패 원인 추적 어려움 | flush.log / handle.log 분리 기록 |
| 버퍼 무한 증가 위험 | MAX_BUFFER_SIZE 초과 시 503 |

<br />

`FOR UPDATE SKIP LOCKED — 단일 프로세서 이상을 위한 설계`

```
현재는 단일 JobProcessor로 운영하지만,
FOR UPDATE SKIP LOCKED 덕분에 나중에 프로세서를 수평 확장해도
동일 job을 두 프로세서가 동시에 가져가는 중복 처리가 DB 레벨에서 방지

지금은 단순하게 단일로 운영하되,
처리량이 부족해지면 프로세서만 늘리면 된다.

단, handle()은 멱등성(idempotency) 보장이 필수다.
(done 업데이트 직전 프로세스가 죽으면, 재기동 시 동일 job이 재처리될 수 있다)
```

4. 트레이드오프 / 리스크 / 배포 전 체크리스트

<br />

`트레이드오프 (의도한 것)`

```
[즉시 202 accepted 응답]
· 클라이언트는 수집 성공으로 인식하지만, 실제 DB 반영은 flush 시점이다.
· 수집 API 특성상 허용되는 구조다.

[flush 단계 재시도 없음]
· DB/네트워크 장애를 버퍼로 무한 흡수하지 않는다.
· 장애가 지속되면 버퍼가 한도까지 차고 503으로 거절된다. (정상 반응)
· 실패 건은 flush.log에만 기록한다. 운영자가 로그로 조치한다.

[단건 fallback의 효과 범위]
· 벌크 INSERT 실패 원인이 DB 연결 장애라면, 단건 fallback도 전부 실패한다.
· 단건 fallback이 실질적으로 효과를 발휘하는 시나리오는
  특정 payload가 DB 제약 조건(유니크, NOT NULL 등)을 위반하는 경우로 한정된다.
· 불량 건만 골라내어 정상 건은 반영하는 것이 목적이며, 재시도 메커니즘이 아니다.

[JobProcessor 단건 순차 처리]
· 처리량보다 예측 가능성을 우선한다.
· 복잡한 동시성 제어를 배제한다.

[트랜잭션 실패 시 Safety Polling 주기만큼 처리 지연]
· processPending() 내 트랜잭션 에러 발생 시 루프를 탈출한다.
· listenClient와 processorPrisma는 별도 커넥션이므로, DB가 복구되어도
  LISTEN 신호가 즉시 재도달하지 않을 수 있다.
· 이 경우 Safety Polling 주기(SAFETY_POLLING_INTERVAL_MS, 기본값 5분)가
  동작할 때까지 최대 해당 주기만큼 pending이 쌓인다.
· 허용 가능한 지연으로 간주하며, 더 짧은 지연이 필요하면
  SAFETY_POLLING_INTERVAL_MS 환경변수로 주기를 단축한다 (재배포 없이 조정 가능).
```

<br />

`리스크 (알고 감수하는 것)`

```
[메모리 버퍼 유실 가능성]
· 비정상 종료(SIGTERM 등)는 onApplicationShutdown에서 flush로 완화한다.
· kill -9 / 하드웨어 장애는 shutdown 훅 자체가 실행되지 않아 버퍼 유실 가능성이 있다.
· WAL 파일로 보완: enqueue 시점에 wal.log에 선기록하고, flush 성공 시 flushed로 표시한다.
  재기동 시 onApplicationBootstrap에서 미처리(pending) 라인을 읽어 버퍼에 복구한다.
· WAL 복구 건수가 MAX_BUFFER_SIZE를 초과하는 경우, 초과분은 버려지고 경고 로그가 남는다.
  초과분이 자주 발생한다면 MAX_BUFFER_SIZE 상향 또는 인스턴스 증설을 검토한다.
· WAL 파일 자체가 손상되는 극단적 상황은 막을 수 없다. (디스크 장애 등)

[DB/네트워크 장애 시 버퍼 적체]
· 한도(MAX_BUFFER_SIZE) 도달 시 503 반환이 의도된 동작이다.
· 버퍼가 계속 찬다면 DB를 살리는 것이 우선이다.

[외부 API 장애 시 failed 누적]
· 자동 재처리 없음. 수동 확인 프로세스가 필요하다.

[handle() 완료 직전 프로세스 종료]
· done 업데이트 전 종료되면 재기동 시 재처리될 수 있다.
· handle() 구현 시 멱등성(idempotency)이 반드시 보장되어야 한다.

[종료 시 처리 중인 handle() 완료 대기]
· onApplicationShutdown에서 isProcessing이 true인 동안 대기 후 커넥션을 해제한다.
· 대기 없이 Prisma 커넥션을 먼저 닫으면 진행 중인 handle()이 에러로 튈 수 있다.
```

<br />

`배포 전 체크리스트`

```
[flush.log / handle.log 모니터링 체계 수립]
· flush.log에 신규 라인 발생 시 즉시 알림 (정상 운영에서는 발생하지 않아야 함)
· handle.log 누적 건수가 1시간 내 N건 초과 시 알림 (N은 운영 합의)
· tail -f 또는 Filebeat 등으로 실시간 감시 권장

[버퍼 임계치 정책 확정]
· MAX_BUFFER_SIZE 운영값 합의 (기본값: 10,000건 / 환경변수 MAX_BUFFER_SIZE로 외부 주입 가능)
· 설정 전 건당 평균 payload 크기를 측정한다.
  (건수 × 평균 크기)가 Node.js --max-old-space-size의 10~20% 이내인지 확인한다.
  예시) payload 평균 1KB × 10,000건 = 약 10MB -> 안전
       payload 평균 50KB × 10,000건 = 약 500MB -> MAX_BUFFER_SIZE 축소 필요
· 503 반환 시 클라이언트 재시도 정책 합의
  예시) 지수 백오프(Exponential Backoff) + 최대 재시도 횟수 설정

[서버 시간존 설정 확인]
· scheduleDailyAt / scheduleWeeklyAt은 서버 로컬 시간 기준으로 동작한다.
· 배포 환경에서 TZ 환경변수가 의도한 시간대로 설정되어 있는지 반드시 확인한다.
  예시) TZ=Asia/Seoul
· TZ 미설정 시 UTC 기준으로 동작하여 VACUUM/REINDEX 실행 시각이 의도와 달라질 수 있다.

[handle() 멱등성 검증]
· 동일 payload로 두 번 호출되어도 부작용이 없는지 확인
  예시) 외부 API 호출 전 중복 확인, upsert 사용, 고유 키 방어 등
· 재기동 시 processing -> pending 복구 로직 동작 확인 (onApplicationBootstrap)

[LISTEN 재연결 완전 실패 시 대응]
· MAX_RECONNECT_RETRY 소진 시 process.exit(1)로 프로세스가 종료됨
· PM2 / systemd 등 프로세스 매니저의 자동 재기동 설정이 반드시 되어 있어야 함
  PM2 예시) pm2 start dist/main.js --restart-delay=3000
· 재기동 시 onApplicationBootstrap에서 processing -> pending 자동 복구됨
· 재연결 소진 시점에 Slack 등 알림 발송 연동 권장

[로컬 개발 환경 주의]
· QUEUE_PROCESSOR_DATABASE_URL 미설정 시 DATABASE_URL로 fallback됨
· 두 URL이 동일하면 커넥션 풀 분리 효과가 없다.
· 로컬에서도 가능하면 별도 URL을 설정하거나, 분리 효과가 없음을 인지하고 사용한다.

[SQL 적용 경로 확정 (6절 참고)]
· setup.sql 단독 vs Prisma migrate + 트리거 별도 적용 중 하나만 선택
· 이중 적용 시 인덱스 충돌 발생 가능

[환경변수 확인]
· TZ 설정 여부 확인 (미설정 시 UTC 기준 동작 — VACUUM/REINDEX 시각 의도와 달라질 수 있음)
· EXTERNAL_API_URL 설정 여부 확인 (미설정 시 handle() Step 2에서 빈 URL로 요청됨)
· QUEUE_PROCESSOR_DATABASE_URL 설정 여부 확인 (미설정 시 풀 분리 효과 없음)
· SAFETY_POLLING_INTERVAL_MS 설정 여부 확인 (미설정 시 기본값 300000ms = 5분으로 동작)
  트랜잭션 실패 후 최대 이 주기만큼 pending 처리가 지연될 수 있으므로 운영 정책에 맞게 합의한다
```

```
· setup.sql의 CREATE TABLE / CREATE INDEX 는 실행하지 않음
· notify_new_job 트리거 블록만 psql 등으로 별도 실행
· migrate dev 만으로는 트리거가 생성되지 않음

* 두 경로를 동시에 적용하면 인덱스 이름 충돌 또는 중복 인덱스가 생성된다.
```

<br />

`sql/setup.sql`

```sql
-- job_queue 테이블
CREATE TABLE job_queue (
  id           SERIAL PRIMARY KEY,
  type         VARCHAR     NOT NULL,
  payload      JSONB       NOT NULL,
  status       VARCHAR     NOT NULL DEFAULT 'pending',
  error        TEXT,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  processed_at TIMESTAMPTZ
);

-- 상태 + 생성일 복합 인덱스 (pending 조회 성능)
CREATE INDEX idx_job_queue_status_created
  ON job_queue (status, created_at);

-- INSERT 시 LISTEN/NOTIFY 신호를 발행하는 트리거 함수
CREATE OR REPLACE FUNCTION notify_new_job()
RETURNS trigger AS $$
BEGIN
  PERFORM pg_notify('new_job', 'batch');
  RETURN NULL;
END;
$$ LANGUAGE plpgsql;

DROP TRIGGER IF EXISTS trg_notify_new_job ON job_queue;

-- statement 단위 트리거: 벌크 INSERT여도 신호는 1회만 발행
CREATE TRIGGER trg_notify_new_job
AFTER INSERT ON job_queue
FOR EACH STATEMENT EXECUTE FUNCTION notify_new_job();
```

7. Prisma 설정

<br />

`prisma/schema.prisma`

```prisma
model JobQueue {
  id          Int       @id @default(autoincrement())
  type        String
  payload     Json
  status      String    @default("pending")
  error       String?
  createdAt   DateTime  @default(now()) @map("created_at") @db.Timestamptz(3)
  processedAt DateTime? @map("processed_at") @db.Timestamptz(3)

  @@index([status, createdAt])
  @@map("job_queue")
}
```

```
기존 `schema.prisma`의 `generator` · `datasource`는 유지하고, `JobQueue` 모델 블록만 추가한다.

setup.sql을 이미 적용했다면 `prisma generate` 만 실행한다.
```

<br />

`handle() Step 1에서 사용하는 Contract 모델`

```prisma
// handle()의 upsert 대상 테이블.
// contractNo를 고유 키로 사용해 멱등성을 보장한다.
// 실제 비즈니스 스펙에 맞게 필드를 확장한다.
model Contract {
  id          Int      @id @default(autoincrement())
  contractNo  String   @unique @map("contract_no")
  dbCode      String   @map("db_code")
  data        Json
  createdAt   DateTime @default(now()) @map("created_at") @db.Timestamptz(3)
  updatedAt   DateTime @updatedAt @map("updated_at") @db.Timestamptz(3)

  @@map("contract")
}
```

```
· contractNo에 @unique 제약이 있어야 upsert의 where 조건으로 사용할 수 있다.
· handle()에서 this.prisma.contract.upsert()를 호출하므로
  이 모델이 schema에 없으면 Prisma 타입 에러가 발생한다.
· migrate dev 실행 시 contract 테이블도 함께 생성된다.
```

`queue/queue.buffer.ts`

```ts
import {
  Injectable,
  OnApplicationBootstrap,
  OnApplicationShutdown,
} from '@nestjs/common';
import { PrismaService } from '../prisma.service';
import { EnqueueDto } from './dto/enqueue.dto';
import { AppLogger } from '../common/app-logger';
import { appendJsonLog } from '../common/append-json-log';
import { walAppend, walMarkFlushed, walReadPending, walCleanup } from '../common/wal-log';

@Injectable()
export class QueueBuffer implements OnApplicationBootstrap, OnApplicationShutdown {
  // ─── 설정 ───────────────────────────────────────────────────────────────────
  private readonly FLUSH_INTERVAL_MS = 500;   // 0.5초마다 한 틱
  private readonly BATCH_SIZE = 200;           // 틱당 최대 처리 건수
  // 환경변수로 외부 주입 가능 — 재배포 없이 조정할 수 있다
  // 설정 전 (건수 × 평균 payload 크기)가 --max-old-space-size의 10~20% 이내인지 확인
  private readonly MAX_BUFFER_SIZE =
    Number(process.env.MAX_BUFFER_SIZE) || 10_000;

  // ─── 상태 ───────────────────────────────────────────────────────────────────
  private readonly buffer: Array<{ walId: string; dto: EnqueueDto }> = [];
  private timer?: ReturnType<typeof setInterval>;

  // 현재 틱의 flush가 진행 중인지 여부 (타이머 중복 실행 방지)
  private isFlushing = false;

  // 진행 중인 flush가 완료되면 resolve되는 Promise (종료 시 대기용)
  private flushingDone: Promise<void> = Promise.resolve();

  private readonly logger = new AppLogger(QueueBuffer.name);

  constructor(private readonly prisma: PrismaService) {}

  // ─── 라이프사이클 ────────────────────────────────────────────────────────────

  async onApplicationBootstrap() {
    // kill -9 / 하드웨어 장애로 유실된 버퍼를 WAL에서 복구
    // wal.log에 pending 상태로 남은 라인을 읽어 버퍼에 다시 추가한다
    try {
      const pending = await walReadPending();
      if (pending.length > 0) {
        this.logger.warn(`WAL 복구: ${pending.length}건을 버퍼에 재적재합니다.`);

        for (const { walId, dto } of pending) {
          // MAX_BUFFER_SIZE 초과분은 버리고 경고 로그를 남긴다
          // 초과분이 반복적으로 발생하면 MAX_BUFFER_SIZE 상향 또는 인스턴스 증설을 검토한다
          if (this.buffer.length >= this.MAX_BUFFER_SIZE) {
            this.logger.warn(
              `WAL 복구 중 버퍼 한도(${this.MAX_BUFFER_SIZE}) 초과 — 나머지 ${pending.length - this.buffer.length}건을 버립니다.`,
            );
            break;
          }
          this.buffer.push({ walId, dto });
        }
      }
    } catch (e) {
      this.logger.error('WAL 복구 실패', e instanceof Error ? e.stack : String(e));
    }

    // 0.5초마다 flushChunk 실행
    this.timer = setInterval(() => void this.flushChunk(), this.FLUSH_INTERVAL_MS);
  }

  async onApplicationShutdown() {
    if (this.timer) clearInterval(this.timer);

    // 진행 중인 flush가 있으면 완료될 때까지 대기, 없으면 즉시 통과
    if (this.isFlushing) await this.flushingDone;

    // 버퍼에 남은 데이터 전부 처리
    while (this.buffer.length > 0) {
      await this.flushChunk();
    }

    // flushed 상태로 표시된 라인 정리 (다음 기동 시 재복구 방지)
    try {
      await walCleanup();
    } catch (e) {
      this.logger.error('WAL 정리 실패', e instanceof Error ? e.stack : String(e));
    }
  }

  // ─── 공개 메서드 ─────────────────────────────────────────────────────────────

  /**
   * 버퍼에 요청을 추가한다.
   * enqueue 시점에 wal.log에 선기록해 kill -9 / 하드웨어 장애 시 유실을 방지한다.
   * DB 반영은 타이머 또는 종료 시에만 수행된다.
   * @returns 버퍼에 추가되면 true, 버퍼 초과 시 false
   */
  async enqueue(dto: EnqueueDto): Promise<boolean> {
    if (this.buffer.length >= this.MAX_BUFFER_SIZE) return false;

    // WAL 선기록: DB flush 전에 파일에 먼저 기록해 프로세스 비정상 종료 시 복구 가능하게 한다
    // WAL 기록 실패 시에도 버퍼에는 추가한다 (WAL은 보조 수단이므로 enqueue 자체를 막지 않는다)
    let walId = '';
    try {
      walId = await walAppend(dto);
    } catch (e) {
      this.logger.warn(`WAL 선기록 실패 (버퍼 추가는 계속 진행): ${String(e)}`);
    }

    this.buffer.push({ walId, dto });
    return true;
  }

  // ─── 내부 메서드 ─────────────────────────────────────────────────────────────

  /**
   * 버퍼에서 최대 BATCH_SIZE(200건)를 꺼내 job_queue 테이블에 INSERT한다.
   * · 틱 중복 실행 방지: isFlushing 플래그 사용
   * · 벌크 INSERT 실패 시 단건 fallback으로 성공 가능한 건만 반영
   *   (효과적인 시나리오: 특정 payload가 DB 제약 조건을 위반하는 경우)
   *   (DB 연결 장애 시에는 단건도 전부 실패하며 모두 flush.log에 기록됨)
   * · 단건도 실패한 건은 flush.log에 기록하고 버린다 (재시도 없음)
   * · INSERT 성공한 건은 WAL에서 flushed로 표시한다
   *
   * [walId filter(Boolean) 이유]
   * walId가 빈 문자열('')인 경우는 walAppend()가 실패한 케이스다.
   * WAL에 기록 자체가 없으므로 flushed로 표시할 대상이 없다.
   * filter(Boolean)으로 빈 walId를 제거해 walMarkFlushed()에 전달되지 않도록 한다.
   */
  private async flushChunk(): Promise<void> {
    // 이미 처리 중이거나 버퍼가 비어 있으면 스킵
    if (this.isFlushing || this.buffer.length === 0) return;

    this.isFlushing = true;

    // 종료 훅이 이 Promise를 await해 완료 신호를 기다린다
    let resolveFlushing!: () => void;
    this.flushingDone = new Promise<void>((resolve) => {
      resolveFlushing = resolve;
    });

    try {
      // 버퍼 앞에서 최대 200건 추출
      const chunk = this.buffer.splice(0, this.BATCH_SIZE);
      const successWalIds: string[] = [];

      try {
        // 벌크 INSERT 시도
        await this.prisma.jobQueue.createMany({
          data: chunk.map(({ dto }) => ({
            type: 'DB_GATHER',
            payload: dto,
            status: 'pending',
          })),
        });
        // 전체 성공 시 모든 WAL ID를 flushed로 표시
        // walId가 ''인 것은 walAppend 실패 케이스 — WAL에 기록이 없으므로 filter로 제거
        successWalIds.push(...chunk.map(({ walId }) => walId).filter(Boolean));
      } catch (bulkError) {
        // 벌크 실패 시 단건 fallback: 불량 건만 골라내기 위한 분리 목적
        this.logger.warn(`벌크 INSERT 실패, 단건 fallback 시작 (${chunk.length}건): ${String(bulkError)}`);
        for (const { walId, dto } of chunk) {
          try {
            await this.prisma.jobQueue.create({
              data: { type: 'DB_GATHER', payload: dto, status: 'pending' },
            });
            // 단건 성공 시 해당 WAL ID만 flushed로 표시 (walId가 ''인 경우 제외)
            if (walId) successWalIds.push(walId);
          } catch (singleError) {
            // 단건도 실패한 건만 로그에 기록 (WAL에는 pending 상태로 유지)
            await this.safeWriteFlushLog({
              context: 'QueueBuffer',
              dto,
              bulkError: String(bulkError),
              singleError: String(singleError),
              timestamp: new Date().toISOString(),
            });
          }
        }
      }

      // INSERT 성공한 건들의 WAL 항목을 flushed로 표시
      if (successWalIds.length > 0) {
        try {
          await walMarkFlushed(successWalIds);
        } catch (e) {
          // WAL mark 실패 시 재기동 때 해당 건이 중복 복구될 수 있다
          // job_queue INSERT는 이미 성공했으므로 handle()의 멱등성으로 중복 처리를 방지한다
          this.logger.warn(`WAL flushed 표시 실패 (재기동 시 중복 복구 가능): ${String(e)}`);
        }
      }
    } catch (e) {
      this.logger.error('flush 예외 발생', e instanceof Error ? e.stack : String(e));
    } finally {
      this.isFlushing = false;
      resolveFlushing(); // 종료 훅에 완료 신호 전달
    }
  }

  // flush.log 기록 실패 시 logger.error로 fallback (로그 기록 자체가 에러로 튀면 안 됨)
  private async safeWriteFlushLog(payload: Record<string, unknown>) {
    try {
      await appendJsonLog('flush.log', payload);
    } catch (e) {
      this.logger.error('flush.log 기록 실패', `${String(e)} / payload: ${JSON.stringify(payload)}`);
    }
  }
}
```

```ts
  /**

   * QueueBuffer(배치 INSERT)와 풀을 분리해, 트래픽 폭주 시에도
   * Processor가 커넥션을 확보하지 못해 pending 처리가 밀리는 상황을 방지한다.
   *

   * 단건 순차 처리이므로 3~5개면 충분하다.
   *
   * ※ 클래스 필드 선언 시점에 new PrismaClient()를 하면 ConfigModule이 환경변수를
   *   로딩하기 전이라 process.env 값이 undefined일 수 있다.
   *   onApplicationBootstrap()에서 초기화해 환경변수가 확실히 로딩된 이후에 생성한다.
   * ※ 로컬 개발 시 QUEUE_PROCESSOR_DATABASE_URL 미설정이면 DATABASE_URL로 fallback — 풀 분리 효과 없음
   */

  private cleanupTimer?: ReturnType<typeof setInterval>;
  private safetyPollingTimer?: ReturnType<typeof setInterval>;
  private vacuumTimer?: ReturnType<typeof setTimeout>;
  private reindexTimer?: ReturnType<typeof setTimeout>;
  private reconnectTimer?: ReturnType<typeof setTimeout>;

  constructor(private readonly prisma: PrismaService) {}

  // ─── 라이프사이클 ────────────────────────────────────────────────────────────

  async onApplicationBootstrap() {
    // ConfigModule 로딩 완료 후 환경변수가 확정된 시점에 전용 Prisma 클라이언트 생성
    this.processorPrisma = new PrismaClient({
      datasources: {
        db: {
          url: process.env.QUEUE_PROCESSOR_DATABASE_URL ?? process.env.DATABASE_URL,
        },
      },
    });

    // 비정상 종료로 processing 상태에 걸린 job을 pending으로 복구
    await this.processorPrisma.jobQueue.updateMany({
      where: { status: 'processing' },
      data: { status: 'pending' },
    });

    await this.connectListen();
    await this.processPending(); // 서버 재기동 시 밀린 job 즉시 처리

    // 1시간마다 done 레코드 정리
    this.cleanupTimer = setInterval(() => void this.cleanupDone(), this.CLEANUP_INTERVAL_MS);

    // Safety Polling: SAFETY_POLLING_INTERVAL_MS 주기마다 processPending() 강제 실행 (기본값: 5분)
    // LISTEN 신호를 놓친 경우(재연결 타이밍 gap 등)에도 최대 이 주기 안에 pending이 처리된다
    // isProcessing 가드가 있으므로 LISTEN으로 이미 처리 중이면 자동으로 스킵된다
    // 더 짧은 지연이 필요하면 SAFETY_POLLING_INTERVAL_MS 환경변수로 재배포 없이 조정 가능
    this.safetyPollingTimer = setInterval(
      () => void this.processPending(),
      this.SAFETY_POLLING_INTERVAL_MS,
    );

    // VACUUM ANALYZE: 매일 새벽 4시 (Dead Tuple 정리 + 통계 갱신)
    // scheduleDailyAt / scheduleWeeklyAt은 서버 로컬 시간 기준 — TZ 환경변수 설정 필수
    this.vacuumTimer = this.scheduleDailyAt(4, 0, () => void this.runVacuumAnalyze());

    // REINDEX CONCURRENTLY: 매주 일요일 새벽 3시 (Index Bloat 방지)
    this.reindexTimer = this.scheduleWeeklyAt(0, 3, 0, () => void this.runReindexConcurrently());
  }

  async onApplicationShutdown() {
    this.isShuttingDown = true;
    clearInterval(this.cleanupTimer);
    clearInterval(this.safetyPollingTimer);
    clearTimeout(this.vacuumTimer);
    clearTimeout(this.reindexTimer);
    clearTimeout(this.reconnectTimer);

    // 진행 중인 processPending이 있으면 완료될 때까지 대기
    // 대기 없이 커넥션을 먼저 닫으면 handle() 내부 Prisma 호출이 에러로 튄다
    if (this.isProcessing) await this.processingDone;

    // LISTEN 전용 pg.Client 종료
    if (this.listenClient) {
      this.listenClient.removeAllListeners();
      await this.listenClient.end();
    }

    // 전용 Prisma 클라이언트 종료 (직접 new했으므로 직접 해제)
    // bootstrap 이전에 shutdown이 호출되는 극단적 상황 대비 null 가드
    if (this.processorPrisma) await this.processorPrisma.$disconnect();
  }

  // ─── LISTEN/NOTIFY ──────────────────────────────────────────────────────────

  private async connectListen() {
    this.listenClient = new Client({ connectionString: process.env.DATABASE_URL });
    await this.listenClient.connect();
    await this.listenClient.query('LISTEN new_job');

    // 신호 수신 시 pending job 처리 시작
    this.listenClient.on('notification', () => void this.processPending());

    // 커넥션 에러/종료 시 재연결 시도
    this.listenClient.on('error', (err) => {
      this.logger.warn(`LISTEN 에러: ${err?.message ?? err}`);
      void this.reconnect();
    });
    this.listenClient.on('end', () => {
      this.logger.warn('LISTEN 커넥션 종료됨');
      void this.reconnect();
    });
  }

  private async reconnect() {
    // 재연결 중복 실행 방지 & 종료 중이면 스킵
    if (this.isReconnecting || this.isShuttingDown) return;
    this.isReconnecting = true;

    // 기존 커넥션 정리 (이미 끊긴 경우 에러 무시)
    if (this.listenClient) {
      this.listenClient.removeAllListeners();
      await this.listenClient
        .end()
        .catch((e) => this.logger.warn(`기존 커넥션 종료 중 무시: ${e?.message}`));
    }

    let reconnected = false;
    for (let attempt = 1; attempt <= this.MAX_RECONNECT_RETRY; attempt++) {
      if (this.isShuttingDown) break;

      // 선형 대기: 1초, 2초, 3초, 4초, 5초
      await new Promise<void>((resolve) => {
        this.reconnectTimer = setTimeout(resolve, attempt * 1000);
      });

      try {
        await this.connectListen();
        await this.processPending();
        reconnected = true;
        break;
      } catch (e) {
        this.logger.error(`재연결 실패 (${attempt}/${this.MAX_RECONNECT_RETRY})`, String(e));
      }
    }

    if (!reconnected && !this.isShuttingDown) {
      // 재연결 전부 실패 -> 프로세스 종료 후 PM2/systemd가 재기동하도록 위임
      // 재기동 시 onApplicationBootstrap에서 processing -> pending 자동 복구
      // TODO: 이 시점에 Slack 등 알림 발송 연동 권장
      this.logger.error(
        `LISTEN 재연결 ${this.MAX_RECONNECT_RETRY}회 전부 실패. 프로세스를 종료합니다.`,
      );
      process.exit(1);
    }

    this.isReconnecting = false;
  }

  // ─── Job 처리 ────────────────────────────────────────────────────────────────

  /**
   * pending job을 하나씩 꺼내 처리한다.
   * · FOR UPDATE SKIP LOCKED: 다른 프로세서가 가져간 row는 건너뜀
   * · pending이 없어지면 루프를 즉시 탈출하고 다음 LISTEN 신호를 기다린다
   *   (polling 방식처럼 계속 SELECT하면 LISTEN 기반 설계 의도와 모순이므로 break 사용)
   * · isProcessing 가드: notification이 연속으로 와도 루프는 항상 1개만 실행된다
   * · processingDone: onApplicationShutdown에서 커넥션 해제 전 완료를 기다린다
   *
   * [트랜잭션 실패 후 처리 재개 타이밍]
   * processorPrisma 커넥션과 listenClient 커넥션은 별도이므로,
   * DB가 잠깐 끊겼다 복구되어도 LISTEN 신호가 즉시 재도달하지 않을 수 있다.
   * 이 경우 Safety Polling이 동작할 때까지 최대 SAFETY_POLLING_INTERVAL_MS(기본값: 5분)간 pending이 쌓인다.
   * 더 짧은 지연이 필요하면 SAFETY_POLLING_INTERVAL_MS 환경변수를 줄인다 (재배포 없이 조정 가능).
   */
  private async processPending() {
    // 이미 처리 중이거나 종료 중이면 스킵
    // notification이 짧은 시간에 여러 번 와도 루프 중복 실행을 방지한다
    if (this.isShuttingDown || this.isProcessing) return;
    this.isProcessing = true;

    // 종료 훅이 이 Promise를 await해 handle() 완료 후 커넥션을 해제할 수 있도록 한다
    let resolveProcessing!: () => void;
    this.processingDone = new Promise<void>((resolve) => {
      resolveProcessing = resolve;
    });

    try {
      while (true) {
        if (this.isShuttingDown) break;

        try {
          const job = await this.processorPrisma.$transaction(
            async (tx) => {
              // FOR UPDATE SKIP LOCKED: 처리 중인 row는 건너뜀 (멀티 프로세서 안전)
              const rows = await tx.$queryRaw<{ id: number; payload: unknown }[]>`
                SELECT id, payload
                FROM job_queue
                WHERE status = 'pending'
                ORDER BY created_at ASC
                LIMIT 1
                FOR UPDATE SKIP LOCKED
              `;

              if (rows.length === 0) return null;

              // 락 획득 성공 -> processing으로 상태 전환
              await tx.jobQueue.update({
                where: { id: rows[0].id },
                data: { status: 'processing' },
              });

              return rows[0];
            },
            { timeout: 10_000 }, // 트랜잭션 타임아웃 10초 (SELECT + UPDATE 1건만 포함)
          );

          // pending 소진 → 루프 탈출 후 다음 LISTEN 신호를 기다린다
          // (polling 방식처럼 계속 SELECT하면 LISTEN 기반 설계 의도와 모순)
          if (!job) break;

          try {
            await this.handle(job.payload as JobPayload);

            // 비즈니스 처리 성공 -> done
            await this.processorPrisma.jobQueue.update({
              where: { id: job.id },
              data: { status: 'done', processedAt: new Date() },
            });
          } catch (e) {
            // 비즈니스 처리 실패 -> failed + 로그
            const error = e instanceof Error ? e.message : String(e);
            await this.processorPrisma.jobQueue.update({
              where: { id: job.id },
              data: { status: 'failed', error, processedAt: new Date() },
            });
            await this.safeWriteHandleLog({
              context: 'QueueProcessor',
              jobId: job.id,
              payload: job.payload,
              error,
              timestamp: new Date().toISOString(),
            });
          }
        } catch (transactionError) {
          // DB 커넥션 끊김 등 트랜잭션 자체 실패
          // listenClient의 error/end 이벤트가 reconnect()를 트리거할 것을 기대하지만,
          // processorPrisma 커넥션은 별도이므로 여기서도 에러를 기록하고 루프를 탈출한다.
          // Safety Polling(SAFETY_POLLING_INTERVAL_MS 주기) 또는 다음 LISTEN 신호가 올 때까지 처리가 중단된다.
          this.logger.error('트랜잭션 오류', String(transactionError));
          await new Promise((resolve) => setTimeout(resolve, 1000));
          break;
        }
      }
    } finally {
      this.isProcessing = false;
      resolveProcessing(); // 종료 훅에 완료 신호 전달
    }
  }

  // ─── 비즈니스 로직 ───────────────────────────────────────────────────────────

  /**
   * 실제 비즈니스 처리 로직.
   *
   * ① 멱등성(idempotency) 보장
   *    done 업데이트 전 프로세스가 종료되면 재기동 시 동일 job이 재처리될 수 있다.
   *    → upsert로 동일 contractNo가 두 번 들어와도 중복 행 없이 최신 값으로 덮어쓴다.
   *
   * ② 외부 API 타임아웃 (AbortController)
   *    handle()이 hang되면 QueueProcessor 전체가 멈춘다.
   *    Promise.race는 타임아웃 후에도 원본 Promise가 백그라운드에서 계속 실행되므로 사용하지 않는다.
   *
   * [처리 순서]
   *    Step 1. 수신 데이터를 실제 DB 테이블에 upsert
   *    Step 2. 외부 API로 전송
   */
  private async handle(payload: JobPayload): Promise<void> {
    const TIMEOUT_MS = 10_000; // 외부 API 타임아웃 10초

    // ── Step 1. 수신 데이터를 실제 DB 테이블에 upsert ────────────────────────
    // contractNo를 고유 키로 사용해 멱등성 보장
    // 동일 job이 재처리되더라도 중복 행 없이 최신 값으로 덮어쓴다
    await this.prisma.contract.upsert({
      where: { contractNo: payload.contractNo },
      update: {
        dbCode: payload.dbCode,
        data: payload.payload,
        updatedAt: new Date(),
      },
      create: {
        contractNo: payload.contractNo,
        dbCode: payload.dbCode,
        data: payload.payload,
      },
    });

    // ── Step 2. 외부 API로 전송 ───────────────────────────────────────────────
    // AbortController로 타임아웃 설정 — hang 시 전체 프로세서가 멈추는 것을 방지
    const controller = new AbortController();
    const timeoutId = setTimeout(() => controller.abort(), TIMEOUT_MS);

    try {
      const res = await fetch(process.env.EXTERNAL_API_URL ?? '', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({
          contractNo: payload.contractNo,
          dbCode: payload.dbCode,
          payload: payload.payload,
        }),
        signal: controller.signal, // 타임아웃 시 요청 자체가 즉시 취소됨
      });

      if (!res.ok) {
        // 4xx: 요청 자체가 잘못됨 — 재시도해도 동일하게 실패하므로 에러로 처리
        // 5xx: 외부 서버 장애 — 마찬가지로 에러로 처리 후 failed 기록
        throw new Error(`외부 API 오류: ${res.status} ${res.statusText}`);
      }
    } finally {
      clearTimeout(timeoutId); // 정상 완료 시에도 타이머 반드시 해제
    }
  }

  // ─── 정리 / 유지보수 ─────────────────────────────────────────────────────────

  /** done 레코드 중 24시간이 지난 것을 삭제한다 (1시간 주기) */
  private async cleanupDone() {
    try {
      const cutoff = new Date(Date.now() - this.RETENTION_MS);
      await this.prisma.jobQueue.deleteMany({
        where: { status: 'done', createdAt: { lt: cutoff } },
      });
    } catch (e) {
      // 정리 실패는 운영에 치명적이지 않으므로 로그만 남기고 계속 진행
      // 다음 주기(1시간 후)에 재시도된다
      this.logger.error('[cleanup] done 레코드 정리 실패', String(e));
    }
  }

  /**
   * VACUUM ANALYZE job_queue
   *
   * 대량 INSERT/DELETE 반복 테이블은 autovacuum이 따라가지 못해
   * Dead Tuple이 누적되고 쿼리 성능이 저하될 수 있다.
   * VACUUM ANALYZE로 Dead Tuple 정리 + 쿼리 플래너 통계를 갱신한다.
   * 실행 중 테이블 잠금 없음 -> 운영 중 안전하게 실행 가능.
   *
   * 실행 시각은 서버 로컬 시간 기준 (TZ 환경변수에 따라 결정됨)
   */
  private async runVacuumAnalyze() {
    try {
      this.logger.log('[maintenance] VACUUM ANALYZE 시작');
      // Prisma는 VACUUM을 트랜잭션으로 감싸지 않아야 하므로 $executeRawUnsafe 사용
      await this.prisma.$executeRawUnsafe('VACUUM ANALYZE job_queue');
      this.logger.log('[maintenance] VACUUM ANALYZE 완료');
    } catch (e) {
      this.logger.error('[maintenance] VACUUM ANALYZE 실패', String(e));
    } finally {
      // 다음 날 새벽 4시에 재예약
      if (!this.isShuttingDown) {
        this.vacuumTimer = this.scheduleDailyAt(4, 0, () => void this.runVacuumAnalyze());
      }
    }
  }

  /**
   * REINDEX INDEX CONCURRENTLY idx_job_queue_status_created
   *
   * pending -> done -> DELETE 사이클이 반복되면 인덱스 내부에
   * 사용되지 않는 페이지(Bloat)가 쌓여 인덱스 스캔 성능이 저하된다.
   *
   * CONCURRENTLY 키워드: 잠금 없이 백그라운드 재구성 (PostgreSQL 12+)
   * CONCURRENTLY 없는 REINDEX는 ShareLock -> 운영 중 INSERT/SELECT 블로킹 -> 절대 사용 금지
   * REINDEX CONCURRENTLY는 트랜잭션 블록 안에서 실행 불가 -> $executeRawUnsafe 사용
   *
   * 실행 시각은 서버 로컬 시간 기준 (TZ 환경변수에 따라 결정됨)
   */
  private async runReindexConcurrently() {
    try {
      this.logger.log('[maintenance] REINDEX INDEX CONCURRENTLY 시작');
      await this.prisma.$executeRawUnsafe(
        'REINDEX INDEX CONCURRENTLY idx_job_queue_status_created',
      );
      this.logger.log('[maintenance] REINDEX INDEX CONCURRENTLY 완료');
    } catch (e) {
      this.logger.error('[maintenance] REINDEX 실패', String(e));
    } finally {
      // 다음 주 일요일 새벽 3시에 재예약
      if (!this.isShuttingDown) {
        this.reindexTimer = this.scheduleWeeklyAt(0, 3, 0, () => void this.runReindexConcurrently());
      }
    }
  }

  // ─── 스케줄러 헬퍼 ───────────────────────────────────────────────────────────

  /**
   * 매일 특정 시각(hour:minute)에 콜백을 한 번 실행하는 타이머를 예약한다.
   * setInterval 대신 setTimeout 체인을 사용해 실행 시각의 정확도를 높인다.
   * 시각 계산은 서버 로컬 시간 기준 — TZ 환경변수(예: TZ=Asia/Seoul)가 반드시 설정되어야 한다.
   */
  private scheduleDailyAt(hour: number, minute: number, cb: () => void): ReturnType<typeof setTimeout> {
    const now = new Date();
    const next = new Date(now);
    next.setHours(hour, minute, 0, 0);
    // 오늘 해당 시각이 이미 지났으면 내일로
    if (next <= now) next.setDate(next.getDate() + 1);
    return setTimeout(cb, next.getTime() - now.getTime());
  }

  /**
   * 매주 특정 요일(0=일, 1=월, …)·시각에 콜백을 한 번 실행하는 타이머를 예약한다.
   * scheduleDailyAt과 동일하게 날짜 비교로 처리해 당일 시각이 아직 안 지난 경우를 정확히 처리한다.
   * (% 7 || 7 방식은 오늘 해당 요일이고 시각이 아직 안 지났을 때 다음 주로 밀리는 버그가 있다)
   * 시각 계산은 서버 로컬 시간 기준 — TZ 환경변수(예: TZ=Asia/Seoul)가 반드시 설정되어야 한다.
   */
  private scheduleWeeklyAt(
    dayOfWeek: number,
    hour: number,
    minute: number,
    cb: () => void,
  ): ReturnType<typeof setTimeout> {
    const now = new Date();
    const next = new Date(now);
    next.setHours(hour, minute, 0, 0);
    const daysUntil = (dayOfWeek - now.getDay() + 7) % 7;
    next.setDate(next.getDate() + daysUntil);
    // 같은 요일이고 해당 시각이 이미 지났으면 다음 주로
    if (next <= now) next.setDate(next.getDate() + 7);
    return setTimeout(cb, next.getTime() - now.getTime());
  }

  // ─── 로그 ────────────────────────────────────────────────────────────────────

  private async safeWriteHandleLog(payload: Record<string, unknown>) {
    try {
      await appendJsonLog('handle.log', payload);
    } catch (e) {
      this.logger.error('handle.log 기록 실패', `${String(e)} / payload: ${JSON.stringify(payload)}`);
    }
  }
}
```

```ts
/**
 * 날짜별 디렉터리(logs/YYYY/MM/DD/)에 JSON 한 줄을 append한다.

  fileName: string,
  payload: Record<string, unknown>,
): Promise<void> {
  const now = new Date();
  const y = now.getFullYear();
  const m = String(now.getMonth() + 1).padStart(2, '0');
  const d = String(now.getDate()).padStart(2, '0');
  const dir = join(process.cwd(), 'logs', `${y}`, `${m}`, `${d}`);
  await mkdir(dir, { recursive: true });
  await appendFile(join(dir, fileName), `${JSON.stringify(payload)}\n`, 'utf8');
}
```

`common/wal-log.ts`

```ts
import { appendFile, readFile, mkdir } from 'node:fs/promises';
import { existsSync } from 'node:fs';
import { join } from 'node:path';
import { randomUUID } from 'node:crypto';
import { EnqueueDto } from '../queue/dto/enqueue.dto';

// WAL 파일 경로: 재기동 시 날짜가 바뀌어도 동일 파일을 읽어야 하므로 고정 경로 사용
const WAL_PATH = join(process.cwd(), 'logs', 'wal.log');

interface WalEntry {
  walId: string;
  dto?: EnqueueDto;
  status: 'pending' | 'flushed' | 'compacted';
  timestamp?: string;
}

/**
 * enqueue 시점에 WAL에 선기록 (항상 파일 끝에 추가)
 * @returns 생성된 walId (flush 성공 시 flushed 표시에 사용)
 */
export async function walAppend(dto: EnqueueDto): Promise<string> {
  const walId = randomUUID();
  const entry: WalEntry = {
    walId,
    dto,
    status: 'pending',
    timestamp: new Date().toISOString(),
  };

  await mkdir(join(process.cwd(), 'logs'), { recursive: true });
  // 파일 전체를 건드리지 않고 끝에 한 줄만 추가하여 I/O 부하 최소화
  await appendFile(WAL_PATH, JSON.stringify(entry) + '\n', 'utf8');
  return walId;
}

/**
 * DB INSERT 성공한 건들을 flushed로 표시 (수정 대신 완료 표식을 추가)
 * 기존 파일을 읽고 쓸 필요가 없어 성능이 비약적으로 향상됨
 */
export async function walMarkFlushed(walIds: string[]): Promise<void> {
  if (walIds.length === 0) return;

  const marks = walIds
    .map((id) => JSON.stringify({ walId: id, status: 'flushed' }))
    .join('\n') + '\n';

  await appendFile(WAL_PATH, marks, 'utf8');
}

/**
 * 재기동 시 pending 상태 항목을 읽어 버퍼 복구에 사용
 * 파일을 순차적으로 읽으며 Map을 통해 처리 완료된 항목을 걸러냄
 * compacted 표식을 만나면 그 이전 항목은 전부 처리 완료로 간주하고 Map을 초기화한다
 */
export async function walReadPending(): Promise<Array<{ walId: string; dto: EnqueueDto }>> {
  if (!existsSync(WAL_PATH)) return [];

  const raw = await readFile(WAL_PATH, 'utf8');
  const lines = raw.split('\n').filter(Boolean);
  
  const pendingMap = new Map<string, EnqueueDto>();

  for (const line of lines) {
    try {
      const entry = JSON.parse(line) as WalEntry;
      if (entry.status === 'compacted') {
        // 정상 종료 시 walCleanup()이 기록한 표식
        // 이 시점까지의 모든 항목은 이미 처리 완료 — Map을 초기화한다
        pendingMap.clear();
      } else if (entry.status === 'pending' && entry.dto) {
        pendingMap.set(entry.walId, entry.dto);
      } else if (entry.status === 'flushed') {
        // flushed 표식이 있으면 pending Map에서 제거
        pendingMap.delete(entry.walId);
      }
    } catch (e) {
      // 파싱 에러 발생 시 해당 라인 스킵
      continue;
    }
  }

  return Array.from(pendingMap.entries()).map(([walId, dto]) => ({ walId, dto }));
}

/**
 * 정상 종료 시, 컴팩션 완료 표식을 파일 끝에 한 줄 추가한다.
 * walReadPending()은 이 표식 이전의 모든 라인을 이미 처리된 것으로 간주한다.
 *
 * writeFile로 전체 재작성하지 않으므로:
 * · 재작성 도중 프로세스가 죽어 파일이 유실되는 문제가 없다.
 * · 디스크가 꽉 찬 상태에서도 기존 데이터가 손상되지 않는다.
 * · I/O 비용이 최소화된다 (한 줄 append만 수행).
 *
 * 파일이 장기간 누적되어 크기가 커지는 경우는 주기적 재기동(배포) 시
 * walReadPending()의 pending 결과가 0건이면 빈 파일로 교체해도 안전하다.
 */
export async function walCleanup(): Promise<void> {
  await mkdir(join(process.cwd(), 'logs'), { recursive: true });
  // 컴팩션 완료 표식을 파일 끝에 append
  // walReadPending()은 이 라인 이전의 모든 항목을 compacted(처리 완료)로 간주한다
  await appendFile(WAL_PATH, JSON.stringify({ status: 'compacted', timestamp: new Date().toISOString() }) + '\n', 'utf8');
}
```

```ts
import { ConsoleLogger } from '@nestjs/common';
import { appendFile, mkdir } from 'node:fs/promises';
import { join } from 'node:path';

/**
 * NestJS 기본 ConsoleLogger를 확장한 커스텀 Logger.
 *
 * · 콘솔 출력은 그대로 유지 (기존 동작 변경 없음)
 * · 동시에 logs/YYYY/MM/DD/app.log 에 텍스트 한 줄로 기록
 * · NestFactory.create() 에 주입하면 NestJS 내부 로그까지 모두 파일로 남는다
 *
 * 사용법:
 *   // main.ts
 *   const app = await NestFactory.create(AppModule, { logger: new AppLogger() });
 *
 *   // 각 클래스 내부
 *   private readonly logger = new AppLogger(QueueBuffer.name);
 *   this.logger.log('버퍼 flush 완료');
 *   this.logger.warn('재연결 시도');
 *   this.logger.error('flush 실패', error.stack);
 */
export class AppLogger extends ConsoleLogger {
  // 파일 기록 실패가 연속으로 발생할 때 콘솔이 도배되지 않도록 에러 쿨다운
  private static lastFileErrorAt = 0;
  private static readonly FILE_ERROR_COOLDOWN_MS = 60_000; // 1분

  override log(message: string, context?: string) {
    super.log(message, context);
    void this.writeToFile('LOG', message, context);
  }

  override warn(message: string, context?: string) {
    super.warn(message, context);
    void this.writeToFile('WARN', message, context);
  }

  override error(message: string, trace?: string, context?: string) {
    super.error(message, trace, context);
    // trace(스택)가 있으면 함께 기록
    const body = trace ? `${message}\n${trace}` : message;
    void this.writeToFile('ERROR', body, context);
  }

  override debug(message: string, context?: string) {
    super.debug(message, context);
    void this.writeToFile('DEBUG', message, context);
  }

  override verbose(message: string, context?: string) {
    super.verbose(message, context);
    void this.writeToFile('VERBOSE', message, context);
  }

  // logs/YYYY/MM/DD/app.log 에 한 줄 append
  private async writeToFile(level: string, message: string, context?: string): Promise<void> {
    try {
      const now = new Date();
      const y = now.getFullYear();
      const m = String(now.getMonth() + 1).padStart(2, '0');
      const d = String(now.getDate()).padStart(2, '0');
      const dir = join(process.cwd(), 'logs', `${y}`, `${m}`, `${d}`);

      await mkdir(dir, { recursive: true });

      // 형식: 2025-05-03T04:12:00.000Z [ERROR] [QueueProcessor] 트랜잭션 오류: ...
      const ctx = context ? ` [${context}]` : '';
      const line = `${now.toISOString()} [${level}]${ctx} ${message}\n`;

      await appendFile(join(dir, 'app.log'), line, 'utf8');

      // 파일 기록 성공 시 에러 쿨다운 초기화
      AppLogger.lastFileErrorAt = 0;
    } catch (e) {
      // 파일 기록 실패는 콘솔에만 남기고 서버를 죽이지 않는다
      // 쿨다운으로 동일 에러가 1분에 한 번만 출력되도록 제한
      const now = Date.now();
      if (now - AppLogger.lastFileErrorAt > AppLogger.FILE_ERROR_COOLDOWN_MS) {
        AppLogger.lastFileErrorAt = now;
        process.stderr.write(`[AppLogger] app.log 기록 실패: ${String(e)}\n`);
      }
    }
  }
}
```

10. DB 유지보수 및 정리 전략

| 대상 | 방식 | 주기 |
| --- | --- | --- |
| done 레코드 | 자동 삭제 (24시간 보존) | 1시간마다 |
| failed 레코드 | 수동 확인 후 삭제 | — |
| Dead Tuple | `VACUUM ANALYZE job_queue` | 매일 새벽 4시 (서버 로컬 시간 기준) |
| Index Bloat | `REINDEX INDEX CONCURRENTLY` | 매주 일요일 새벽 3시 (서버 로컬 시간 기준) |

<br />

```
[Dead Tuple (VACUUM ANALYZE)]
· INSERT/DELETE가 반복되면 PostgreSQL MVCC 구조상 삭제된 행이
  즉시 제거되지 않고 Dead Tuple로 남는다.
· autovacuum이 기본 처리하지만 대량 트래픽 시 따라가지 못할 수 있다.
· VACUUM ANALYZE는 Dead Tuple 정리 + 쿼리 플래너 통계 갱신을 동시에 수행한다.
· 실행 중 테이블 잠금 없음 — 운영 중 안전.

[Index Bloat (REINDEX CONCURRENTLY)]
· pending -> done -> DELETE 사이클이 반복되면 인덱스 내부에
  사용되지 않는 페이지(Bloat)가 쌓여 인덱스 스캔 성능이 저하된다.
· REINDEX INDEX CONCURRENTLY는 잠금 없이 백그라운드에서 재구성한다 (PostgreSQL 12+).
· REINDEX INDEX (CONCURRENTLY 없음)는 ShareLock 발생 — 절대 사용하지 않는다.

[실행 시각 기준]
· scheduleDailyAt / scheduleWeeklyAt은 서버 로컬 시간 기준이다.
· TZ=Asia/Seoul 설정 시 한국 표준시(KST) 기준 새벽 4시 / 새벽 3시에 실행된다.
· TZ 미설정 시 UTC 기준으로 동작하므로 배포 환경의 TZ 설정을 반드시 확인한다.
```

11. 로그 구조 및 모니터링

```
logs/
└── YYYY/MM/DD/
    ├── app.log      # 전체 서버 로그 (log / warn / error)
    ├── flush.log    # 배치 INSERT 최종 실패 건 (정상 운영에서는 발생하지 않아야 함)
    └── handle.log   # handle() 실패 건 (외부 연동/후속 처리 오류)
logs/
└── wal.log          # enqueue 선기록 (날짜별 아닌 고정 경로 — 재기동 시 복구용)
```

<br />

```
[app.log]
· 전체 서버 로그 (log / warn / error)
· WAL 복구 발생 시 WARN 레벨로 복구 건수가 기록된다
  ("WAL 복구: N건을 버퍼에 재적재합니다.")
· WAL 복구 중 MAX_BUFFER_SIZE 초과 시 WARN 레벨로 초과 건수가 기록된다
  (초과분이 반복되면 MAX_BUFFER_SIZE 상향 또는 인스턴스 증설 검토)

[flush.log]
· 벌크 INSERT 실패 후 단건 INSERT도 실패한 건
· 신규 라인 발생 즉시 알림 권장
· 동일 dbCode / contractNo 반복 실패 시 데이터 품질 이슈 의심

[handle.log]
· QueueProcessor handle() 실패 건
· 1시간 내 누적 N건 초과 시 알림 (N은 운영 합의 필요)
· 동일 jobId 반복 기록 시 멱등성 구현 점검 필요

[wal.log]
· enqueue 시점에 선기록되는 WAL 파일 (logs/ 루트 고정 경로)
· status: pending — DB에 아직 반영되지 않은 항목
· status: flushed — DB INSERT 성공 후 표시된 항목
· status: compacted — 정상 종료 시 walCleanup()이 기록하는 표식
  이 표식 이전의 모든 항목은 재기동 시 처리 완료로 간주되어 복구 대상에서 제외된다
· 재기동 시 pending 항목이 남아 있으면 WAL 복구 로그가 app.log에 기록됨
· pending 항목이 반복적으로 복구된다면 flush 실패 원인 확인 필요
· 날짜별 디렉터리가 아닌 logs/wal.log 고정 경로 사용
  (재기동 시 날짜가 바뀌어도 동일 파일을 읽어야 복구가 가능하기 때문)
· 파일은 append-only로만 기록되므로 장기간 운영 시 크기가 늘어날 수 있다
  재기동(배포) 후 pending 복구 건수가 0이면 파일을 수동으로 비워도 안전하다
```
