# 대량 INSERT를 안정적으로 처리하기 위한 구조 갖추기

<br />
<br />

* 적은 리소스를 사용해서 대량의 Insert를 처리하는 방법을 알아보자

---

```
개발자는 항상 트레이드오프를 해야 한다.

완전하고 충분한 환경은 없기에,
항상 최적화를 고려하고 적은 리소스를 최대로 사용해야 한다.

싱글 스레드 환경에서 가장 적은 리소스로 대용량 Insert 처리해보자.
```

1. 핵심은 버퍼링, 배치, 큐이다.

```
아주 낮은 사양의 인스턴스가 있다고 가정해보자.

대량의 Insert를 처리할 수 있을까?

방법은 최대한 모으고 (버퍼링),
과정을 생략하고,
조금씩 가져와서 처리하는 것이다. (배치 + 큐)

여기서는 인스턴스의 리소스를 최대한으로 활용하기 위해
가벼운 노드와 백엔드 프레임워크인 NestJs를 사용한다.
ORM은 Prisma를 사용한다.
```


3. 전체 흐름

```
1. 이벤트 랜딩에서 DB Gathering (클라이언트는 API Insert 요청)

2. Controller -> Service 메모리 버퍼에 push (즉시 응답 -> 트레이드오프)

3. 버퍼 200건 or 0.5초마다 -> Batch INSERT -> job_queue 테이블
   flush 실패 시 retryCount 추적, 3회 초과 dead letter 격리 + 로그
   (DB 트리거가 INSERT 감지 -> pg_notify('new_job') 발송)

4. Worker가 LISTEN으로 신호 수신 -> pending 없어질 때까지 루프 처리
   handle() 실패 시 failed 상태로 기록 + dead-letter.log 기록
   커넥션 끊기면 exponential backoff 재연결 + processPendingAll()

5. 1시간마다 done job 배치 삭제 (24시간 보존)
   failed job은 영구 보존 -> 퇴근 전 수동 확인 + 처리 후 삭제
```

4. 왜 이런 구조인가?

| 문제 | 해결 |
|------|------|
| 건당 INSERT -> DB 왕복 비용 | 버퍼링 + Batch INSERT |
| 후속 처리 부하가 요청 타임에 집중 | Queue로 분리 -> Worker가 순서대로 처리 |
| Worker 여러 대 중복 처리 | FOR UPDATE SKIP LOCKED |
| polling -> 무의미한 쿼리 낭비 | LISTEN/NOTIFY -> 신호 왔을 때만 처리 |
| LISTEN 커넥션 단절 시 silent failure | exponential backoff 자동 재연결 |
| 불량 데이터가 flush를 영원히 블로킹 | retryCount 3회 초과 dead letter 격리 |
| handle() 외부 API 장애 | failed 기록 + 퇴근 전 수동 처리 (외부 장애는 자동 재처리 의미없음) |
| handle() 무한 대기 | timeout 5초 설정 |
| burst 트래픽 시 큐 적체 | 신호 수신 시 pending 소진될 때까지 루프 |
| 쿠버네티스 스케일아웃 시 중복 처리 | isProcessing 플래그 제거, DB 락만 신뢰 |
| done job 테이블 무한 증가 | 1시간마다 done 배치 삭제 (24시간 보존) |
| failed job 원인 파악 불가 | 영구 보존 + dead-letter.log 별도 기록 |

<br />

5. 트레이드오프 / 리스크 / 크리티컬

<br />

`트레이드오프 - 의도한 것`

```
0.5초치 메모리 버퍼 유실 허용
- 건당 INSERT 제거가 목표였기 때문에 감수
- 비정상 종료 시 최대 0.5초치 유실
- 정상 종료(배포, 재시작)는 onApplicationShutdown flush로 보장
- DB 서버가 안정적이니 flush 실패 자체가 희박

즉시 응답 (accepted)
- 클라이언트는 성공으로 알지만 아직 DB에 없는 상태
- 데이터 수집 특성상 허용한 트레이드오프
- flush 실패는 retryCount + dead letter로 이중 대응

handle() 자동 재처리 없음
- 외부 API(보험사) 장애는 우리가 해결 불가
- 자동 재처리가 의미없는 상황
- failed 영구 보존 후 퇴근 전 수동 확인이 더 효율적
- 원인 파악 + 보험사 문의 근거 자료로 활용

handle() 처리량이 병목
- 외부 API 연동이 건당 느리면 처리량 저하
- 구조 문제가 아닌 handle() 최적화 문제
- pending 적체는 큐의 정상 동작 범위
```

<br />

`리스크 - 알고 감수하는 것`

```
스케일아웃 시
- FOR UPDATE SKIP LOCKED로 중복 처리는 방지
- 쿠버네티스 CPU 70% 기준 스케일아웃 대응 가능

flush retryCount 기준 미검증
- flush dead letter: 3회
- 운영 데이터 보고 튜닝 필요
```

<br />

`크리티컬 - 배포 전 필수`

```
dead letter 알림 미구축
- 일단 롤링 로그로 대응
- 추후 Slack 연동 권장
- 배포 전 dead-letter.log 모니터링 체계 확보 필수
```

7. SQL 설정

<br />

`sql/setup.sql`

```sql
-- job_queue 테이블
CREATE TABLE job_queue (
  id           SERIAL PRIMARY KEY,
  type         VARCHAR   NOT NULL,
  payload      JSONB     NOT NULL,
  status       VARCHAR   NOT NULL DEFAULT 'pending',
  error        TEXT,
  created_at   TIMESTAMP NOT NULL DEFAULT NOW(),
  processed_at TIMESTAMP
);

-- Worker polling용 인덱스
-- status + created_at 복합 인덱스 -> pending 조회 + FIFO 정렬 최적화
-- INSERT 성능에 거의 영향 없음
CREATE INDEX idx_job_queue_status_created
  ON job_queue (status, created_at);

-- LISTEN/NOTIFY 트리거
-- job_queue에 INSERT 될 때마다 'new_job' 채널로 신호 발송
CREATE OR REPLACE FUNCTION notify_new_job()
RETURNS trigger AS $$
BEGIN
  PERFORM pg_notify('new_job', NEW.id::text);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_notify_new_job
AFTER INSERT ON job_queue
FOR EACH ROW EXECUTE FUNCTION notify_new_job();
```

`gather.service.ts`

```ts
import {
  Injectable,
  OnApplicationBootstrap,
  OnApplicationShutdown,
  Inject,
} from '@nestjs/common';
import { WINSTON_MODULE_PROVIDER } from 'nest-winston';
import { Logger as WinstonLogger } from 'winston';
import { DbGatherDto } from '../dto/db-gather.dto';
import { PrismaService } from '../prisma.service';

/**
 * 메모리 버퍼링 + Batch INSERT 서비스
 *
 * 왜 버퍼링을 하냐?
 * - 건당 INSERT는 매번 DB 왕복 -> 트래픽 몰리면 커넥션 풀 고갈
 * - 버퍼에 모았다가 한번에 INSERT -> DB 부하 대폭 감소
 *
 * flush 조건 (둘 중 하나라도 충족되면)
 * 1. 버퍼가 BUFFER_LIMIT 이상 쌓였을 때 (즉시)
 * 2. FLUSH_INTERVAL 시간이 지났을 때 (타이머)
 *
 * 트레이드오프
 * - 비정상 종료 시 최대 0.5초치 버퍼 유실 허용
 * - 정상 종료는 onApplicationShutdown flush로 보장
 * - DB 서버가 안정적이니 flush 실패 자체가 희박
 *
 * 불량 데이터 격리
 * - flush 실패 시 retryCount 추적
 * - MAX_RETRY 초과 시 dead letter 로그 기록 후 격리
 * - 블로킹 없이 뒤 데이터 계속 처리 가능
 */

interface BufferItem {
  dto: DbGatherDto;
  retryCount: number;
}

@Injectable()
export class GatherService implements OnApplicationBootstrap, OnApplicationShutdown {
  // 메모리 버퍼 (싱글톤이라 앱 내내 같은 인스턴스 -> 요청 간 데이터 유지됨)
  private buffer: BufferItem[] = [];
  private timer: NodeJS.Timeout;

  private readonly FLUSH_INTERVAL = 500; // 0.5초마다 자동 flush
  private readonly BUFFER_LIMIT = 200;   // 200건 쌓이면 즉시 flush
  private readonly MAX_RETRY = 3;        // flush 최대 재시도 횟수

  constructor(
    private readonly prisma: PrismaService,
    @Inject(WINSTON_MODULE_PROVIDER)
    private readonly logger: WinstonLogger,
  ) {}

  /**
   * 앱이 완전히 뜬 후 한 번만 호출됨 (NestJS 라이프사이클 훅)
   * constructor에서 시작하면 모듈이 덜 뜬 상태일 수 있어서 여기서 시작
   */
  onApplicationBootstrap() {
    // 타이머는 요청과 무관하게 딱 한 번만 시작
    // 요청마다 시작하면 타이머가 중복 생성됨
    this.timer = setInterval(() => this.flush(), this.FLUSH_INTERVAL);
    this.logger.info('버퍼 타이머 시작', { context: 'GatherService' });
  }

  /**
   * 앱 종료 직전 호출됨 (NestJS 라이프사이클 훅)
   * main.ts에서 app.enableShutdownHooks() 필수
   * 없으면 배포/재시작 시 버퍼에 남은 데이터 유실
   */
  onApplicationShutdown() {
    clearInterval(this.timer);
    this.flush();
    this.logger.info('버퍼 타이머 종료', { context: 'GatherService' });
  }

  /**
   * Controller에서 호출 (await 안 함)
   * 버퍼에 넣기만 하고 즉시 리턴 -> Controller가 바로 응답 가능
   */
  enqueue(dto: DbGatherDto) {
    this.buffer.push({ dto, retryCount: 0 });

    // 200건 넘으면 타이머 기다리지 않고 즉시 flush
    if (this.buffer.length >= this.BUFFER_LIMIT) {
      this.flush();
    }
  }

  private async flush() {
    if (this.buffer.length === 0) return;

    // 현재 버퍼 스냅샷 뜨고 비우기
    // splice로 한번에 처리해야 flush 도중 들어온 데이터 안 잃음
    const items = this.buffer.splice(0, this.buffer.length);

    try {
      // Prisma createMany -> 진짜 SQL 한방 (bulk insert)
      // INSERT INTO job_queue (type, payload, status) VALUES (...), (...), ...
      await this.prisma.jobQueue.createMany({
        data: items.map(({ dto }) => ({
          type: 'DB_GATHER',
          payload: dto as object,
          status: 'pending',
        })),
      });

      this.logger.info(`Flush 완료: ${items.length}건`, { context: 'GatherService' });
    } catch (e) {
      this.logger.warn(`Flush 실패: ${e.message}`, { context: 'GatherService' });

      const retryItems: BufferItem[] = [];

      for (const item of items) {
        const next = { ...item, retryCount: item.retryCount + 1 };

        if (next.retryCount >= this.MAX_RETRY) {
          // Dead letter: MAX_RETRY 초과 -> 격리 + 로그
          // dead-letter.log에 error 레벨로 기록됨
          // 추후 Slack 연동 시 이 로그 기준으로 알림 발송
          this.logger.error('flush dead letter', {
            context: 'GatherService',
            retryCount: next.retryCount,
            dto: next.dto,
            timestamp: new Date().toISOString(),
          });
        } else {
          retryItems.push(next);
        }
      }

      // 재시도 대상만 버퍼 앞에 다시 삽입
      if (retryItems.length > 0) {
        this.buffer.unshift(...retryItems);
      }
    }
  }
}
```

```ts
import {
  Injectable,
  OnApplicationBootstrap,
  OnApplicationShutdown,
  Inject,
} from '@nestjs/common';
import { Client } from 'pg';
import { WINSTON_MODULE_PROVIDER } from 'nest-winston';
import { Logger as WinstonLogger } from 'winston';
import { PrismaService } from '../prisma.service';

/**
 * 큐 Worker
 *
 * 왜 큐를 쓰냐?
 * - INSERT 부하와 후속 처리 부하를 분리하기 위함
 * - 트래픽이 몰려도 INSERT만 빠르게 받고, 처리는 순서대로 여유있게
 * - 가공 로직, 외부 API 연동 등 무거운 작업일수록 큐가 빛남
 *
 * LISTEN/NOTIFY 방식
 * - DB 트리거가 job_queue INSERT 시 pg_notify('new_job') 발송
 * - Worker가 신호 수신 시 pending 소진될 때까지 루프 처리
 * - polling (1초마다 무조건 쿼리) 대비 리소스 낭비 없음
 *
 * 자동 재연결
 * - pg.Client 단절 시 exponential backoff로 재연결 (최대 30초)
 * - 재연결 후 processPendingAll()로 놓친 job 처리
 * - isReconnecting 플래그로 중복 재연결 방지
 *
 * 스케일아웃 대응
 * - isProcessing 플래그 없음
 * - FOR UPDATE SKIP LOCKED만 신뢰
 * - 쿠버네티스 CPU 70% 스케일아웃 시에도 중복 처리 없음
 *
 * handle() 실패 처리
 * - 외부 API(보험사) 장애는 자동 재처리 의미없음
 * - failed 영구 보존 후 퇴근 전 수동 확인 + 처리
 * - 원인 파악 + 보험사 문의 근거 자료로 활용
 *
 * 주의: LISTEN은 반드시 전용 커넥션이어야 함
 * - Prisma 커넥션 풀에서 꺼내쓰면 안 됨
 * - 풀이 커넥션을 재사용/반납하기 때문에 LISTEN이 끊김
 */
@Injectable()
export class GatherWorker implements OnApplicationBootstrap, OnApplicationShutdown {
  private listenClient: Client;
  private isReconnecting = false;       // reconnect 중복 방지
  private cleanupTimer: NodeJS.Timeout; // done 삭제 타이머 (1시간)

  private readonly CLEANUP_INTERVAL = 60 * 60 * 1000;  // 1시간
  private readonly RETENTION_MS = 24 * 60 * 60 * 1000; // 24시간 보존
  private readonly HANDLE_TIMEOUT = 5000;               // handle() timeout 5초

  constructor(
    private readonly prisma: PrismaService,
    @Inject(WINSTON_MODULE_PROVIDER)
    private readonly logger: WinstonLogger,
  ) {}

  async onApplicationBootstrap() {
    await this.connectWithRetry();

    // 앱 시작 시 밀린 pending job 처리 (서버 재시작 대비)
    await this.processPendingAll();

    // done 삭제 타이머 (1시간마다)
    // failed는 삭제하지 않음 -> 퇴근 전 수동 확인 + 처리 후 삭제
    this.cleanupTimer = setInterval(() => this.cleanup(), this.CLEANUP_INTERVAL);

    this.logger.info('Worker LISTEN 시작', { context: 'GatherWorker' });
  }

  async onApplicationShutdown() {
    clearInterval(this.cleanupTimer);
    try {
      await this.listenClient.end();
    } catch {}
    this.logger.info('Worker LISTEN 종료', { context: 'GatherWorker' });
  }

  /**
   * LISTEN 전용 커넥션 연결 + 이벤트 핸들러 등록
   * exponential backoff: 1초 -> 2초 -> 4초 -> ... -> 최대 30초
   * 재귀 대신 while 루프로 스택 오버플로우 방지
   */
  private async connectWithRetry() {
    let retries = 0;

    while (true) {
      try {
        this.listenClient = new Client({
          connectionString: process.env.DATABASE_URL,
        });
        await this.listenClient.connect();
        await this.listenClient.query('LISTEN new_job');

        // 신호 수신 시 pending 소진될 때까지 루프 처리
        // isProcessing 플래그 없음 -> FOR UPDATE SKIP LOCKED만 신뢰
        // 쿠버네티스 스케일아웃 시에도 중복 처리 없음
        this.listenClient.on('notification', async () => {
          await this.processPendingAll();
        });

        // 커넥션 끊기면 자동 재연결
        this.listenClient.on('error', async (err) => {
          this.logger.warn(`LISTEN 커넥션 에러: ${err.message}`, { context: 'GatherWorker' });
          await this.reconnect();
        });
        this.listenClient.on('end', async () => {
          this.logger.warn('LISTEN 커넥션 종료됨', { context: 'GatherWorker' });
          await this.reconnect();
        });

        this.logger.info('LISTEN 커넥션 연결 완료', { context: 'GatherWorker' });
        break; // 성공하면 루프 탈출

      } catch (e) {
        const delay = Math.min(1000 * 2 ** retries, 30000); // 최대 30초
        this.logger.warn(`LISTEN 연결 실패, ${delay}ms 후 재시도 (${retries + 1}회)`, {
          context: 'GatherWorker',
        });
        await this.sleep(delay);
        retries++;
      }
    }
  }

  /**
   * 재연결
   * isReconnecting 플래그로 중복 재연결 방지
   * 재연결 후 processPendingAll()로 끊긴 동안 놓친 job 처리
   */
  private async reconnect() {
    if (this.isReconnecting) return;
    this.isReconnecting = true;

    try {
      await this.listenClient.end();
    } catch {}

    await this.connectWithRetry();

    // 재연결 중 놓쳤을 수 있는 pending job 처리
    await this.processPendingAll();

    this.isReconnecting = false;
  }

  // pending 없어질 때까지 루프
  private async processPendingAll() {
    let hasMore = true;
    while (hasMore) {
      hasMore = await this.processNext();
    }
  }

  // 처리했으면 true, 없으면 false 반환
  private async processNext(): Promise<boolean> {
    return await this.prisma.$transaction(async (tx) => {
      // FOR UPDATE SKIP LOCKED
      // - FOR UPDATE: 가져온 row에 락 걸기
      // - SKIP LOCKED: 이미 락 걸린 row는 건너뜀
      // -> Worker 여러 대 돌아도 같은 job 중복 처리 안 됨
      // -> Prisma는 FOR UPDATE SKIP LOCKED 미지원 -> $queryRaw 사용
      const jobs = await tx.$queryRaw<{ id: number; payload: any }[]>`
        SELECT id, payload
        FROM job_queue
        WHERE status = 'pending'
        ORDER BY created_at ASC
        LIMIT 1
        FOR UPDATE SKIP LOCKED
      `;

      if (jobs.length === 0) return false;

      const job = jobs[0];

      // processing으로 변경 (다른 Worker가 못 가져가게)
      await tx.jobQueue.update({
        where: { id: job.id },
        data: { status: 'processing' },
      });

      try {
        // timeout 5초 -> 외부 API 무응답 시 Worker 무한 대기 방지
        await Promise.race([
          this.handle(job.payload),
          new Promise((_, reject) =>
            setTimeout(() => reject(new Error('handle timeout')), this.HANDLE_TIMEOUT),
          ),
        ]);

        await tx.jobQueue.update({
          where: { id: job.id },
          data: {
            status: 'done',
            processedAt: new Date(),
          },
        });
      } catch (e) {
        // 외부 API 장애는 자동 재처리 의미없음
        // failed 영구 보존 -> 퇴근 전 수동 확인 + 처리
        // dead-letter.log에 error 레벨로 기록됨
        this.logger.error('handle failed', {
          context: 'GatherWorker',
          jobId: job.id,
          payload: job.payload,
          error: e.message,
          timestamp: new Date().toISOString(),
        });

        await tx.jobQueue.update({
          where: { id: job.id },
          data: {
            status: 'failed',
            error: e.message,
            processedAt: new Date(),
          },
        });
      }

      return true;
    });
  }

  /**
   * done job 정리 (1시간마다 실행)
   * 24시간 지난 done job만 삭제
   *
   * failed job은 삭제하지 않음
   * -> 영구 보존으로 원인 파악 + 보험사 문의 근거 자료 활용
   * -> 퇴근 전 수동 확인 후 처리 완료 시 직접 DELETE
   */
  private async cleanup() {
    const cutoff = new Date(Date.now() - this.RETENTION_MS);

    const result = await this.prisma.jobQueue.deleteMany({
      where: {
        status: 'done', // failed는 건드리지 않음
        createdAt: { lt: cutoff },
      },
    });

    this.logger.info(`정리 완료: ${result.count}건 삭제`, { context: 'GatherWorker' });
  }

  /**
   * 실제 비즈니스 로직
   * - 보험 데이터 가공
   * - 실제 테이블에 저장
   * - 외부 API 연동 등
   * timeout 5초 적용 중 (HANDLE_TIMEOUT)
   */
  private async handle(payload: any) {
    this.logger.info(`처리중: ${JSON.stringify(payload)}`, { context: 'GatherWorker' });
    // TODO: 실제 처리 로직
  }

  private sleep(ms: number) {
    return new Promise((resolve) => setTimeout(resolve, ms));
  }
}
```

10. polling vs LISTEN/NOTIFY 비교

| 항목 | 기존 (polling) | 개선 (LISTEN/NOTIFY) |
|------|--------------|-------------------|
| DB 쿼리 | 1초마다 무조건 | 신호 왔을 때만 |
| 반응 속도 | 최대 1초 지연 | 거의 실시간 |
| 리소스 | sleep마다 쿼리 낭비 | idle 시 0 쿼리 |
| 커넥션 | 풀 사용 | 전용 1개 추가 |
| 단절 대응 | 없음 | exponential backoff 재연결 |
| burst 대응 | 단건 처리 | pending 소진될 때까지 루프 |
| 스케일아웃 | isProcessing 플래그 충돌 | FOR UPDATE SKIP LOCKED만 신뢰 |

11. job 정리 전략

| status | 정리 방식 |
|--------|----------|
| done | 1시간마다 자동 삭제 (24시간 보존) |
| failed | 영구 보존 -> 퇴근 전 수동 확인 + 처리 후 직접 DELETE |

<br />

`퇴근 전 확인용 쿼리`

```sql
-- failed job 전체 확인
-- error 컬럼으로 원인 파악
-- payload로 어떤 데이터인지 확인
SELECT id, type, payload, error, processed_at
FROM job_queue
WHERE status = 'failed'
ORDER BY created_at ASC;

-- 원인 파악 + 처리 완료 후 삭제
DELETE FROM job_queue
WHERE status = 'failed'
AND id = :id;

-- 전체 failed 일괄 삭제 (모두 처리 완료 시)
DELETE FROM job_queue
WHERE status = 'failed';
```
