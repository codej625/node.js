# 대량 INSERT를 안정적으로 처리하기 위한 구조 갖추기

<br />
<br />

* 적은 리소스를 사용해서 대량의 Insert를 처리하는 방법

---

```
개발자는 항상 트레이드오프를 대비해야 한다.

완전하고 충분한 환경은 없기에,
항상 최적화를 고려하고 적은 리소스를 최대로 사용할 수 있는 능력이 중요하다.
```

1. 핵심은 버퍼링과 큐이다.

```
아주 낮은 사양의 인스턴스가 있다고 가정해보자.

대량의 Insert를 처리할 수 있을까?

방법은 Insert를 최대한 모으고 (버퍼링)
가볍게 만들어 조금씩 처리하는 것이다. (큐)

예제는 인스턴스의 리소스를 최대한으로 활용하기 위해
잦은 I/O 작업에서 가볍고 빠른 노드를 사용한다.

NestJs (Framework) + Prisma (ORM)
```

2. 시나리오

```
클라이언트가 네이버에 메인 지면인
스페셜 DA를 구매했다.

랜딩페이지를 만들었지만

(하지만 나중을 위해 스케일아웃 정도는 대비하는 코드를 준비)
```

3. 전체 흐름

```
1. 이벤트 랜딩에서 DB Gathering (클라이언트는 API Insert 요청)

2. Controller -> Service 메모리 버퍼에 push
   성공 시 즉시 accepted 응답, 버퍼 초과 시 503 반환 (Backpressure)

3. 버퍼 200건 or 0.5초마다 -> Batch INSERT -> job_queue 테이블
   flush 실패 시 fallback 분리 삽입(배치 -> 청크 -> 단건)
   최종 실패한 단건만 retryCount 추적, 3회 초과 dead letter 격리 + 로그
   + Prisma 에러 코드 포함 기록
   (DB 트리거가 INSERT 감지 -> pg_notify('new_job') 발송, statement 단위 1회)

4. Worker가 LISTEN으로 신호 수신 -> pending 없어질 때까지 루프 처리
   handle() 실패 시 failed 상태로 기록 + handle-failed.log 기록
   커넥션 끊기면 exponential backoff 재연결 + processPendingAll()
   Heartbeat 실패 N회 연속 시 reconnect() 트리거

5. 1시간마다 done job 배치 삭제 (24시간 보존)
   failed job은 영구 보존 -> 퇴근 전 수동 확인 + 처리 후 삭제
```

4. 전체 프로세스 순서도

```
════════════════════════════════════════════════════
  요청 수신
════════════════════════════════════════════════════

클라이언트
  │
  │  POST /api/db-gather
  ▼
GatherController
  │  enqueue(dto)         <- await 안 함
  │  성공: { status: 'accepted' } / 실패: 503 Service Unavailable
  ▼
GatherService.enqueue()
  │
  ├─▶ 메모리 버퍼에 push { dto, retryCount: 0 }
  └─▶ MAX_BUFFER_SIZE 초과 시 false 반환 (수집 거부)


════════════════════════════════════════════════════
  버퍼 -> job_queue (0.5초마다 or 200건마다)
════════════════════════════════════════════════════

setInterval (0.5초)
  │
  ▼
flush()
  │
  ├─ 버퍼 비어있으면 종료
  │
  ├─ isFlushing=true면 flushRequested=true로 표시 후 종료
  │
  ├─ splice()로 버퍼 전체 꺼냄
  │    └─ 꺼내는 동안 새로 들어온 데이터는 버퍼 뒤에 계속 쌓임
  │
  ▼
createMany() bulk INSERT -> job_queue
  │
  ├─ 성공 ─────────────────────────────────────────▶ 다음 단계로
  │
  └─ 실패
       │
       ├─ split fallback (청크 삽입 -> 단건 삽입)
       │    ├─ 성공한 건 -> 즉시 job_queue 반영
       │    └─ 끝까지 실패한 단건만 retryCount +1
       │
       └─ 단건 retryCount >= 3 -> dead-letter.log 기록 후 격리
                                { context: GatherService, dto: {...}, prismaErrorCode: 'Pxxxx' }


════════════════════════════════════════════════════
  job_queue -> Worker (신호 기반)
════════════════════════════════════════════════════

job_queue INSERT 성공
  │
  ▼
DB 트리거 (trg_notify_new_job)
  │  pg_notify('new_job', 'batch')
  ▼
LISTEN 전용 커넥션 (pg.Client)
  │
  ├─ 정상 -> notification 이벤트 수신
  │        + heartbeat(SELECT 1) 주기 실행
  │
  └─ 단절 -> exponential backoff 재연결 (1초->2초->4초->최대 30초)
            재연결 후 processPendingAll() 로 놓친 job 처리


════════════════════════════════════════════════════
  Worker 처리 (pending 없어질 때까지 루프)
════════════════════════════════════════════════════

processPendingAll()
  │
  ├─ 이미 drain 중이면 drainRequested=true 표시 후 즉시 종료
  │
  ├─ 1회 drain 최대 200건 처리
  │   처리 후 setImmediate로 이벤트 루프 양보
  │
  ▼
processNext() <- 루프
  │
  ├─ pending job 없음 -> 루프 종료
  │
  ▼
[트랜잭션] FOR UPDATE SKIP LOCKED
  │  pending job 1건 가져옴
  │  status -> processing
  │  트랜잭션 종료 (락 해제)
  │
  ▼
[트랜잭션 밖] Promise.race([handle(payload), timeout(5초)])
  │
  ├─ 성공
  │    └─ status -> done
  │
  └─ 실패 (외부 API 장애 or timeout)
       │
       ├─ status -> failed
       ├─ error 컬럼에 에러 메시지 저장
       └─ handle-failed.log 기록
          { context: GatherWorker, jobId: N, error: '...' }


════════════════════════════════════════════════════
  정리 (타이머)
════════════════════════════════════════════════════

cleanupTimer (1시간마다)
  │
  ├─ done + createdAt 24시간 초과 -> DELETE
  │
  └─ failed -> 건드리지 않음 (영구 보존)


════════════════════════════════════════════════════
  퇴근 전 수동 처리
════════════════════════════════════════════════════

handle-failed.log 확인
  │
  ▼
SELECT * FROM job_queue WHERE status = 'failed'
  │
  ├─ error 컬럼으로 원인 파악
  ├─ payload로 어떤 데이터인지 확인
  │
  ▼
처리 완료 후
DELETE FROM job_queue WHERE status = 'failed' AND id = :id


════════════════════════════════════════════════════
  로그 파일 정리
════════════════════════════════════════════════════

app.log           -> 서버 전체 동작 기록 (info + warn + error)
dead-letter.log   -> flush 3회 실패 (DB 장애 의심, jobId 없음)
handle-failed.log -> handle() 실패 (외부 API 장애 의심, jobId 있음)
```

5. 왜 이런 구조인가?

| 문제 | 해결 |
|------|------|
| 건당 INSERT -> DB 왕복 비용 | 버퍼링 + Batch INSERT |
| 후속 처리 부하가 요청 타임에 집중 | Queue로 분리 -> Worker가 순서대로 처리 |
| Worker 여러 대 중복 처리 | FOR UPDATE SKIP LOCKED |
| polling -> 무의미한 쿼리 낭비 | LISTEN/NOTIFY -> 신호 왔을 때만 처리 |
| LISTEN 커넥션 단절 시 silent failure | exponential backoff 자동 재연결 |
| 불량 데이터가 flush를 영원히 블로킹 | retryCount 3회 초과 dead letter 격리 |
| handle() 외부 API 장애 | failed 기록 + 퇴근 전 수동 처리 (외부 장애는 자동 재처리 의미없음) |
| handle() 무한 대기 | timeout 5초 설정 |
| burst 트래픽 시 큐 적체 | 신호 수신 시 pending 소진될 때까지 루프 |
| 쿠버네티스 스케일아웃 시 중복 처리 | isProcessing 플래그 제거, DB 락만 신뢰 |
| done job 테이블 무한 증가 | 1시간마다 done 배치 삭제 (24시간 보존) |
| failed job 원인 파악 불가 | 영구 보존 + 로그 파일 분리 (dead-letter.log / handle-failed.log) |
| 비정상 종료 시 processing 고착 job | 재시작 시 processing -> pending 자동 복구 |
| 트랜잭션 내 handle() 실패 시 failed rollback | 락 획득과 handle() 호출을 트랜잭션 분리 |
| Worker 처리량 확장 요구 | 안정성 우선으로 단건 순차 처리 유지 (Concurrency 확장 제외) |
| 배치 내 불량 1건으로 정상건 동반 실패 | fallback 분리 삽입으로 불량 단건만 재시도/격리 |
| LISTEN/NOTIFY 신호 과다 | FOR EACH STATEMENT 트리거로 신호량 축소 |
| 버퍼 무한 증가로 OOM 가능 | MAX_BUFFER_SIZE 초과 시 503 반환(backpressure) |
| LISTEN 유휴 커넥션 silent death | heartbeat 실패 N회 연속 시 reconnect() |
| 워커 루프 독점으로 API 지연 | drain당 최대 200건 + setImmediate 양보 |
| 종료 중 무한 재연결 시도 | connectWithRetry while 루프에 isShuttingDown 체크 추가 |
| 잘못된 요청이 버퍼까지 도달 | ValidationPipe + class-validator DTO 검증 |

6. 트레이드오프 / 리스크 / 크리티컬

<br />

`트레이드오프 - 의도한 것`

```
0.5초치 메모리 버퍼 유실 허용
- 건당 INSERT 제거가 목표였기 때문에 감수
- 비정상 종료 시 최대 0.5초치 유실
- 정상 종료(배포, 재시작)는 onApplicationShutdown flush로 보장
- DB 서버가 안정적이니 flush 실패 자체가 희박
```

```
즉시 응답 (accepted)
- 클라이언트는 성공으로 알지만 아직 DB에 없는 상태
- 데이터 수집 특성상 허용한 트레이드오프
- flush 실패는 retryCount + dead letter로 이중 대응
```

```
handle() 자동 재처리 없음
- 외부 API(보험사) 장애는 우리가 해결 불가
- 자동 재처리가 의미없는 상황
- failed 영구 보존 후 퇴근 전 수동 확인이 더 효율적
- 원인 파악 + 보험사 문의 근거 자료로 활용
```

```
Worker 단건 순차 처리 유지 (Concurrency 확장 제외)
- 처리량보다 안정성과 예측 가능성을 우선
- LIMIT 1 + FOR UPDATE SKIP LOCKED 기반으로 단순 운영
- Promise.all 기반 병렬 처리 확장은 의도적으로 적용하지 않음
```

```
Poison Pill 단건 격리
- createMany 실패 시 분리 삽입 fallback으로 정상건 우선 반영
- 최종 실패한 단건만 retryCount 증가 및 dead letter 대상
- 1건 불량으로 정상 199건이 함께 격리되지 않도록 보장
```

<br />

`리스크 - 알고 감수하는 것`

```
스케일아웃 시
- FOR UPDATE SKIP LOCKED로 중복 처리는 방지
- 쿠버네티스 CPU 70% 기준 스케일아웃 대응 가능
```

```
flush retryCount 기준 미검증
- flush dead letter: 3회
- 운영 데이터 보고 튜닝 필요
```

```
retryCount 재시작 시 리셋
- retryCount는 메모리 기반이므로 서버 재시작 시 0으로 초기화됨
- 동일 불량 데이터가 재시작 후 다시 유입되면 dead-letter 기준이 리셋됨
- 운영 임계치(MAX_RETRY)를 넉넉히 잡거나, 필요 시 DB 영속화 고려
```

```
handle() 중복 처리 가능성
- handle() 성공 후 update(done) 전 프로세스 종료 시 재처리될 수 있음
- 재시작 시 processing -> pending 복구 로직에 의해 해당 job이 재시도됨
- handle() 구현 시 반드시 멱등성(idempotency) 보장 필요
```

```
fallback 삽입 시 오류 구간 성능 저하
- 정상 구간은 기존 createMany 성능 유지
- 오류 구간에서만 청크/단건 삽입으로 DB 호출 증가
- 대신 정상 데이터 손실/동반 격리 리스크를 크게 줄임
```

<br />

`크리티컬 - 배포 전 필수`

```
dead letter 알림 미구축
- 일단 롤링 로그로 대응
- 추후 Slack 연동 권장
- 배포 전 dead-letter.log / handle-failed.log 모니터링 체계 확보 필수
```

```
LISTEN heartbeat 운영값 확정
- heartbeat 주기(권장 1~5분) 설정
- 실패 연속 횟수 임계치(N) 설정 후 reconnect() 트리거
- heartbeat 중복 실행 방지 플래그(isHeartbeatRunning) 적용
- heartbeat 임계치 도달 시 reconnect() 비동기 트리거로 루프 점유 최소화
- heartbeat 실패 경고 로그 모니터링 필수
```

```
재연결 플래그 고착 방지
- reconnect()는 try/finally로 isReconnecting 해제를 보장
- connect/process 중 예외가 나도 다음 재연결 시도가 막히지 않도록 구성
```

```
flush / handle 실패 로그 관측성 강화
- error message 외 Prisma 에러 코드 함께 기록
- 장애 시 원인 파악 속도 개선
```

```
버퍼 임계치(Backpressure) 운영 기준 확정
- MAX_BUFFER_SIZE 초과 시 수집 거부 정책 적용
- API는 503 Service Unavailable 반환으로 재시도 유도
```

8. SQL 설정

<br />

`sql/setup.sql`

```sql
-- job_queue 테이블
CREATE TABLE job_queue (
  id           SERIAL PRIMARY KEY,
  type         VARCHAR   NOT NULL,
  payload      JSONB     NOT NULL,
  status       VARCHAR   NOT NULL DEFAULT 'pending',
  error        TEXT,
  created_at   TIMESTAMP NOT NULL DEFAULT NOW(),
  processed_at TIMESTAMP
);

-- Worker polling용 인덱스
-- status + created_at 복합 인덱스 -> pending 조회 + FIFO 정렬 최적화
-- INSERT 성능에 거의 영향 없음
CREATE INDEX idx_job_queue_status_created
  ON job_queue (status, created_at);

-- LISTEN/NOTIFY 트리거 (statement 단위)
-- createMany로 여러 건 INSERT 되어도 notify는 1회만 발생
CREATE OR REPLACE FUNCTION notify_new_job()
RETURNS trigger AS $$
BEGIN
  PERFORM pg_notify('new_job', 'batch');
  RETURN NULL;
END;
$$ LANGUAGE plpgsql;

DROP TRIGGER IF EXISTS trg_notify_new_job ON job_queue;

CREATE TRIGGER trg_notify_new_job
AFTER INSERT ON job_queue
FOR EACH STATEMENT EXECUTE FUNCTION notify_new_job();
```

```ts
import { NestFactory } from '@nestjs/core';
import { ValidationPipe } from '@nestjs/common';
import { AppModule } from './app.module';
import { WINSTON_MODULE_NEST_PROVIDER } from 'nest-winston';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  // Winston을 NestJS 기본 로거로 교체
  app.useLogger(app.get(WINSTON_MODULE_NEST_PROVIDER));

  // [추가] 전역 ValidationPipe 등록
  // whitelist: DTO에 없는 필드 자동 제거 (보안)
  // forbidNonWhitelisted: 알 수 없는 필드 있으면 400 반환
  // transform: 요청 body를 DTO 클래스 인스턴스로 자동 변환
  app.useGlobalPipes(
    new ValidationPipe({
      whitelist: true,
      forbidNonWhitelisted: true,
      transform: true,
    }),
  );

  // onApplicationShutdown 훅 활성화 필수
  // 없으면 서버 종료 시 버퍼에 남은 데이터 유실
  app.enableShutdownHooks();

  await app.listen(3000);
}
bootstrap();
```

```ts
import { Module } from '@nestjs/common';
import { WinstonModule } from 'nest-winston';
import * as winston from 'winston';
import 'winston-daily-rotate-file';

// 로그 메시지 상수
// 문자열 하드코딩 대신 상수로 관리 -> 오타 방지 + 필터 조건과 일치 보장
export const LOG_EVENTS = {
  DEAD_LETTER: 'flush dead letter',
  HANDLE_FAILED: 'handle failed',
} as const;

const onlyMessage = (message: string) =>
  winston.format((info) => (info.message === message ? info : false))();

@Module({
  imports: [
    WinstonModule.forRoot({
      transports: [
        // 콘솔 출력
        new winston.transports.Console({
          format: winston.format.combine(
            winston.format.timestamp(),
            winston.format.colorize(),
            winston.format.simple(),
          ),
        }),

        // 일반 로그 (년/월/일 롤링)
        // %DATE% 플레이스홀더를 사용해야 실제 날짜로 치환됨
        // datePattern이 'YYYY/MM/DD'이면 logs/2024/03/15/app.log로 생성
        new winston.transports.DailyRotateFile({
          filename: 'logs/%DATE%/app.log',
          datePattern: 'YYYY/MM/DD',
          zippedArchive: true,
          maxSize: '20m',
          maxFiles: '30d',
          format: winston.format.combine(
            winston.format.timestamp(),
            winston.format.json(),
          ),
        }),

        // flush 실패 전용 로그 (버퍼 -> job_queue INSERT 실패)
```

```ts
import { PrismaClient } from '@prisma/client';

@Injectable()
export class PrismaService extends PrismaClient implements OnModuleInit, OnModuleDestroy {
  constructor() {
    const rawUrl = process.env.DATABASE_URL;
    if (!rawUrl) throw new Error('DATABASE_URL 환경변수가 설정되지 않았습니다.');

    const url = new URL(rawUrl);
    url.searchParams.set('connection_limit', process.env.DATABASE_POOL_SIZE ?? '15');

    super({
      datasources: {
        db: { url: url.toString() },
      },
    });
  }

  async onModuleInit() {
    await this.$connect();
  }

  async onModuleDestroy() {
    await this.$disconnect();
  }
}
```

```ts
  @Post('db-gather')
```

`gather.service.ts`

```ts
import {
  Injectable,
  OnApplicationBootstrap,
  OnApplicationShutdown,
  Inject,
} from '@nestjs/common';
import { WINSTON_MODULE_PROVIDER } from 'nest-winston';
import { Logger as WinstonLogger } from 'winston';
import { DbGatherDto } from '../dto/db-gather.dto';
import { PrismaService } from '../prisma.service';
import { LOG_EVENTS } from '../logger.module';

interface BufferItem {
  dto: DbGatherDto;
  retryCount: number;
}

interface FailedItemMeta {
  item: BufferItem;
  error: string;
  prismaErrorCode: string;
}

@Injectable()
export class GatherService implements OnApplicationBootstrap, OnApplicationShutdown {
  private buffer: BufferItem[] = [];
  private timer: NodeJS.Timeout;
  private isFlushing = false;
  private flushRequested = false;

  private readonly FLUSH_INTERVAL = 500;
  private readonly BUFFER_LIMIT = 200;
  private readonly CHUNK_FALLBACK_SIZE = 20;
  private readonly MAX_RETRY = 3;
  private readonly MAX_BUFFER_SIZE = 10000;

  constructor(
    private readonly prisma: PrismaService,
    @Inject(WINSTON_MODULE_PROVIDER)
    private readonly logger: WinstonLogger,
  ) {}

  // 앱 기동 시 flush 타이머를 시작
  onApplicationBootstrap() {
    this.timer = setInterval(() => {
      void this.flush();
    }, this.FLUSH_INTERVAL);
    this.logger.info('버퍼 타이머 시작', { context: 'GatherService' });
  }

  // 앱 종료 직전 남은 버퍼를 flush해서 유실을 줄임
  async onApplicationShutdown() {
    clearInterval(this.timer);
    await this.flush();
    this.logger.info('버퍼 타이머 종료', { context: 'GatherService' });
  }

  // 버퍼에 적재하고, 임계치 도달 시 즉시 flush를 트리거
  // 버퍼가 꽉 찼으면 false를 반환해 Controller가 503을 응답
  enqueue(dto: DbGatherDto): boolean {
    if (this.buffer.length >= this.MAX_BUFFER_SIZE) {
      this.logger.error('buffer overflow reject', {
        context: 'GatherService',
        bufferLength: this.buffer.length,
        maxBufferSize: this.MAX_BUFFER_SIZE,
        dto,
        timestamp: new Date().toISOString(),
      });
      return false;
    }

    this.buffer.push({ dto, retryCount: 0 });

    if (this.buffer.length >= this.BUFFER_LIMIT) {
      void this.flush();
    }

    return true;
  }

  // 버퍼를 DB에 반영하는 메인 루프
  // 배치 실패 시 fallback 분리 삽입으로 정상건 우선 반영
  private async flush() {
    if (this.isFlushing) {
      this.flushRequested = true;
      return;
    }

    if (this.buffer.length === 0) return;

    this.isFlushing = true;

    try {
      // flush 실행 중 새 flush 요청이 오면 flushRequested=true로 표시
      // 현재 턴 종료 후 do-while이 한 번 더 돌아 누락 없이 이어서 반영함
      do {
        this.flushRequested = false;

        // [수정] splice(0, this.buffer.length) -> splice(0) 으로 간결화
        const items = this.buffer.splice(0);
        if (items.length === 0) break;

        try {
          // 정상 경로 -> 단일 bulk INSERT 시도
          await this.insertBatch(items);

          this.logger.info(`Flush 완료: ${items.length}건`, { context: 'GatherService' });
        } catch (e) {
          const err = e as { message?: string; code?: string };
          const message = err?.message ?? String(e);
          const code = err?.code ?? 'UNKNOWN';

          this.logger.warn(`Flush 배치 실패 -> 분리 삽입 시도: ${message}`, {
            context: 'GatherService',
            prismaErrorCode: code,
            batchSize: items.length,
          });

          // 오류 경로 -> 배치 - 청크 - 단건으로 점진 분해
          // 성공 가능한 데이터부터 먼저 반영
          const failedItems = await this.insertWithFallback(items);
          this.handleFailedItems(failedItems);
        }
      } while (this.flushRequested || this.buffer.length >= this.BUFFER_LIMIT);
    } finally {
      this.isFlushing = false;
    }
  }

  // BufferItem 배열을 job_queue에 배치 삽입
  private async insertBatch(items: BufferItem[]) {
    await this.prisma.jobQueue.createMany({
      data: items.map(({ dto }) => ({
        type: 'DB_GATHER',
        payload: dto as object,
        status: 'pending',
      })),
    });
  }

  // 배치 실패 시 청크 -> 단건 순서로 분해 삽입하여 불량건만 분리
  private async insertWithFallback(items: BufferItem[]): Promise<FailedItemMeta[]> {
    const failedItems: FailedItemMeta[] = [];

    for (let i = 0; i < items.length; i += this.CHUNK_FALLBACK_SIZE) {
      const chunk = items.slice(i, i + this.CHUNK_FALLBACK_SIZE);

      try {
        // 1차 fallback -> 청크 단위 재시도 (성능/분리 균형)
        await this.insertBatch(chunk);
      } catch {
        // 2차 fallback -> 청크도 실패하면 단건으로 내려가
        // 최종 불량 레코드만 식별
        for (const item of chunk) {
          try {
            await this.insertBatch([item]);
          } catch (e) {
            const err = e as { message?: string; code?: string };
            failedItems.push({
              item,
              error: err?.message ?? String(e),
              prismaErrorCode: err?.code ?? 'UNKNOWN',
            });
          }
        }
      }
    }

    return failedItems;
  }

  // 최종 실패한 단건만 retry/dead-letter 처리하고, 재시도 순서를 보존
  private handleFailedItems(items: FailedItemMeta[]) {
    const retryItems: BufferItem[] = [];

    for (const failed of items) {
      const next = { ...failed.item, retryCount: failed.item.retryCount + 1 };

      if (next.retryCount >= this.MAX_RETRY) {
        // [주의] retryCount는 메모리 기반이므로 서버 재시작 시 0으로 리셋됨
        // 동일 불량 데이터가 재시작 후 다시 유입되면 dead-letter 기준이 초기화됨
        // 운영 임계치(MAX_RETRY)를 넉넉히 잡거나, 필요 시 DB 영속화 고려
        this.logger.error(LOG_EVENTS.DEAD_LETTER, {
          context: 'GatherService',
          retryCount: next.retryCount,
          prismaErrorCode: failed.prismaErrorCode,
          error: failed.error,
          dto: next.dto,
          timestamp: new Date().toISOString(),
        });
      } else {
        retryItems.push(next);
      }
    }

    // unshift를 반복 호출하면 순서가 역전됨
    // [수정] 원본 배열 mutate 방지 -> 스프레드로 복사 후 reverse
    if (retryItems.length > 0) {
      this.buffer.unshift(...[...retryItems].reverse());
    }
  }
}
```

`gather.worker.ts`

```ts
import {
  Injectable,
  OnApplicationBootstrap,
  OnApplicationShutdown,
  Inject,
} from '@nestjs/common';
import { Client } from 'pg';
import { WINSTON_MODULE_PROVIDER } from 'nest-winston';
import { Logger as WinstonLogger } from 'winston';
import { PrismaService } from '../prisma.service';
import { LOG_EVENTS } from '../logger.module';

@Injectable()
export class GatherWorker implements OnApplicationBootstrap, OnApplicationShutdown {
  private listenClient: Client;
  private isReconnecting = false;
  private isShuttingDown = false;
  private isDraining = false;
  private drainRequested = false;
  private cleanupTimer: NodeJS.Timeout;
  private heartbeatTimer: NodeJS.Timeout;
  private heartbeatFailCount = 0;
  private isHeartbeatRunning = false;

  private readonly CLEANUP_INTERVAL = 60 * 60 * 1000;
  private readonly RETENTION_MS = 24 * 60 * 60 * 1000;
  private readonly HANDLE_TIMEOUT = 5000;
  private readonly HEARTBEAT_INTERVAL = 60 * 1000; // 1분
  private readonly HEARTBEAT_FAIL_THRESHOLD = 3;   // 3회 연속 실패 시 재연결
  private readonly MAX_DRAIN_BATCH = 200;          // 1회 drain 최대 처리 건수

  constructor(
    private readonly prisma: PrismaService,
    @Inject(WINSTON_MODULE_PROVIDER)
    private readonly logger: WinstonLogger,
  ) {}

  // 워커 시작 -> 고착 job 복구, LISTEN 연결, 초기 drain, heartbeat/cleanup 타이머 시작
  async onApplicationBootstrap() {
    const recovered = await this.prisma.jobQueue.updateMany({
      where: { status: 'processing' },
      data: { status: 'pending' },
    });
    if (recovered.count > 0) {
      this.logger.warn(`processing 고착 job 복구: ${recovered.count}건`, { context: 'GatherWorker' });
    }

    await this.connectWithRetry();
    await this.processPendingAll();
    this.startHeartbeat();

    this.cleanupTimer = setInterval(() => {
      void this.cleanup();
    }, this.CLEANUP_INTERVAL);

    this.logger.info('Worker LISTEN 시작', { context: 'GatherWorker' });
  }

  // 워커 종료 -> 타이머 정리 후 LISTEN 커넥션을 안전하게 종료
  async onApplicationShutdown() {
    this.isShuttingDown = true;
    clearInterval(this.cleanupTimer);
    clearInterval(this.heartbeatTimer);
    try {
      this.listenClient.removeAllListeners();
      await this.listenClient.end();
    } catch {}
    this.logger.info('Worker LISTEN 종료', { context: 'GatherWorker' });
  }

  // LISTEN 전용 커넥션을 백오프로 재시도하며 연결
  private async connectWithRetry() {
    let retries = 0;

    // [수정] while(true) -> while(!this.isShuttingDown)
    // 종료 신호 수신 후에도 무한 재연결을 시도하는 버그 수정
    while (!this.isShuttingDown) {
      try {
        this.listenClient = new Client({
          connectionString: process.env.DATABASE_URL,
        });
        await this.listenClient.connect();
        await this.listenClient.query('LISTEN new_job');

        // notification은 "신호" 역할만 한다
        // 실제 소진 처리는 processPendingAll()이 책임짐
        this.listenClient.on('notification', async () => {
          await this.processPendingAll();
        });

        this.listenClient.on('error', async (err) => {
          this.logger.warn(`LISTEN 커넥션 에러: ${err.message}`, { context: 'GatherWorker' });
          await this.reconnect();
        });
        this.listenClient.on('end', async () => {
          this.logger.warn('LISTEN 커넥션 종료됨', { context: 'GatherWorker' });
          await this.reconnect();
        });

        this.logger.info('LISTEN 커넥션 연결 완료', { context: 'GatherWorker' });
        this.heartbeatFailCount = 0;
        break;
      } catch (e) {
        try {
          await this.listenClient?.end();
        } catch {}

        // [추가] 종료 중이면 재시도 없이 즉시 탈출
        if (this.isShuttingDown) break;

        const message = e instanceof Error ? e.message : String(e);
        const delay = Math.min(1000 * 2 ** retries, 30000);
        this.logger.warn(`LISTEN 연결 실패: ${message}, ${delay}ms 후 재시도 (${retries + 1}회)`, {
          context: 'GatherWorker',
        });
        await this.sleep(delay);
        retries++;
      }
    }
  }

  // 커넥션 이상 시 중복 없이 재연결하고 놓친 pending을 다시 처리
  private async reconnect() {
    if (this.isReconnecting || this.isShuttingDown) return;
    this.isReconnecting = true;

    try {
      try {
        this.listenClient.removeAllListeners();
        await this.listenClient.end();
      } catch {}

      await this.connectWithRetry();
      // 재연결 사이에 쌓였을 수 있는 pending을 즉시 소진
      await this.processPendingAll();
    } catch (e) {
      const message = e instanceof Error ? e.message : String(e);
      this.logger.error(`재연결 처리 중 오류: ${message}`, { context: 'GatherWorker' });
    } finally {
      // connect/process 중 예외가 나도 재연결 플래그가 고착되지 않도록 보장
      this.isReconnecting = false;
    }
  }

  // LISTEN 커넥션 생존 확인용 heartbeat 스케줄러
  // setInterval 콜백은 await를 기다려주지 않으므로,
  // 실제 heartbeat 실행은 runHeartbeatTick()에서 단일 실행 가드로 보호
  private startHeartbeat() {
    this.heartbeatTimer = setInterval(() => {
      void this.runHeartbeatTick();
    }, this.HEARTBEAT_INTERVAL);
  }

  // heartbeat 1회 실행
  // - 중복 tick 방지
  // - 성공 시 실패 카운터 리셋
  // - 연속 실패 임계치 도달 시 reconnect() 트리거
  private async runHeartbeatTick() {
    if (this.isHeartbeatRunning) return;
    if (!this.listenClient || this.isReconnecting || this.isShuttingDown) return;

    this.isHeartbeatRunning = true;
    try {
      await this.listenClient.query('SELECT 1');
      this.heartbeatFailCount = 0;
    } catch (e) {
      this.heartbeatFailCount += 1;
      const message = e instanceof Error ? e.message : String(e);
      this.logger.warn(
        `LISTEN heartbeat 실패 (${this.heartbeatFailCount}/${this.HEARTBEAT_FAIL_THRESHOLD}): ${message}`,
        { context: 'GatherWorker' },
      );

      if (this.heartbeatFailCount >= this.HEARTBEAT_FAIL_THRESHOLD) {
        // silent death 대응 -> error 이벤트가 없어도 heartbeat 누적으로 복구
        this.logger.warn('LISTEN heartbeat 연속 실패로 재연결 시도', {
          context: 'GatherWorker',
        });
        // heartbeat tick이 재연결 완료까지 붙잡히지 않도록 비동기 트리거
        void this.reconnect();
      }
    } finally {
      this.isHeartbeatRunning = false;
    }
  }

  // pending을 drain하는 루프. 중복 진입을 막고 1회당 최대 건수만 처리
  private async processPendingAll() {
    if (this.isDraining) {
      this.drainRequested = true;
      return;
    }

    this.isDraining = true;
    try {
      do {
        this.drainRequested = false;
        let hasMore = true;
        let processed = 0;
        while (hasMore) {
          if (processed >= this.MAX_DRAIN_BATCH) {
            // 한 턴 독점을 방지하기 위해 상한 도달 시 다음 턴으로 이월
            this.drainRequested = true;
            break;
          }

          hasMore = await this.processNext();
          if (hasMore) processed += 1;
        }

        // 이벤트 루프에 제어권을 반환해 HTTP 처리 지연을 완화
        await new Promise<void>((resolve) => setImmediate(resolve));
      } while (this.drainRequested);
    } finally {
      this.isDraining = false;
    }
  }

  // pending 1건을 락으로 가져와 처리/완료/실패 상태로 전환
  private async processNext(): Promise<boolean> {
    // 트랜잭션 1단계 -> 락 획득 + processing 전환만 수행
    // 외부 I/O(handle)는 트랜잭션 밖에서 실행해 락 점유 시간을 최소화
    const job = await this.prisma.$transaction(async (tx) => {
      const jobs = await tx.$queryRaw<{ id: number; payload: unknown }[]>`
        SELECT id, payload
        FROM job_queue
        WHERE status = 'pending'
        ORDER BY created_at ASC
        LIMIT 1
        FOR UPDATE SKIP LOCKED
      `;

      if (jobs.length === 0) return null;

      await tx.jobQueue.update({
        where: { id: jobs[0].id },
        data: { status: 'processing' },
      });

      return jobs[0];
    });

    if (!job) return false;

    try {
      // 트랜잭션 2단계 -> 실제 처리 + timeout 보호
      // [주의] handle() 성공 후 update(done) 전 프로세스 종료 시
      //   재시작 후 해당 job이 processing -> pending으로 복구되어 재처리됨
      //   handle() 구현 시 반드시 멱등성(idempotency) 보장 필요
      await Promise.race([
        this.handle(job.payload),
        new Promise((_, reject) =>
          setTimeout(() => reject(new Error('handle timeout')), this.HANDLE_TIMEOUT),
        ),
      ]);

      await this.prisma.jobQueue.update({
        where: { id: job.id },
        data: {
          status: 'done',
          processedAt: new Date(),
        },
      });
    } catch (e) {
      const message = e instanceof Error ? e.message : String(e);

      this.logger.error(LOG_EVENTS.HANDLE_FAILED, {
        context: 'GatherWorker',
        jobId: job.id,
        payload: job.payload,
        error: message,
        timestamp: new Date().toISOString(),
      });

      await this.prisma.jobQueue.update({
        where: { id: job.id },
        data: {
          status: 'failed',
          // 수동 재처리 판단을 위해 실패 원인을 DB에 남김
          error: message,
          processedAt: new Date(),
        },
      });
    }

    return true;
  }

  // 보존 기간이 지난 done job을 주기적으로 정리
  private async cleanup() {
    const cutoff = new Date(Date.now() - this.RETENTION_MS);

    const result = await this.prisma.jobQueue.deleteMany({
      where: {
        status: 'done',
        createdAt: { lt: cutoff },
      },
    });

    this.logger.info(`정리 완료: ${result.count}건 삭제`, { context: 'GatherWorker' });
  }

  // 실제 비즈니스 처리 지점(외부 API/가공/저장)
  // [주의] 반드시 멱등성(idempotency)을 보장하도록 구현해야 함
  // 이유: handle() 성공 후 update(done) 전 프로세스 종료 시 재처리될 수 있음
  private async handle(payload: unknown) {
    this.logger.info(`처리중: ${JSON.stringify(payload)}`, { context: 'GatherWorker' });
    // TODO: 실제 처리 로직 (멱등성 보장 필수)
  }

  // 재시도/백오프 간격 제어용 유틸 함수
  private sleep(ms: number) {
    return new Promise((resolve) => setTimeout(resolve, ms));
  }
}
```

11. polling vs LISTEN/NOTIFY 비교

| 항목 | 기존 (polling) | 개선 (LISTEN/NOTIFY) |
|------|--------------|-------------------|
| DB 쿼리 | 1초마다 무조건 | 신호 왔을 때만 |
| 반응 속도 | 최대 1초 지연 | 거의 실시간 |
| 리소스 | sleep마다 쿼리 낭비 | idle 시 0 쿼리 |
| 커넥션 | 풀 사용 | 전용 1개 추가 |
| 단절 대응 | 없음 | exponential backoff 재연결 |
| burst 대응 | 단건 처리 | pending 소진될 때까지 루프 |
| 스케일아웃 | isProcessing 플래그 충돌 | FOR UPDATE SKIP LOCKED만 신뢰 |
| 트리거 신호량 | 행 수만큼 신호 발생 가능 | statement 단위 1회 신호 |
| 유휴 커넥션 안정성 | 별도 대응 없음 | heartbeat + 임계치 기반 재연결 |
| 이벤트 루프 공정성 | drain 무제한 수행 | drain당 최대 200건 + 양보 |

```sql
-- payload로 어떤 데이터인지 확인

ORDER BY created_at ASC;

WHERE status = 'failed';
```

<br />

```
로그 종류:

app.log
- 서버 전체 동작 기록
- flush 완료 N건, Worker 처리 중, 재연결 등
- 전체 흐름 추적할 때 확인

dead-letter.log
- flush 실패 3회 초과 시 기록 (GatherService)
- job_queue에 못 들어간 것 (jobId 없음)
- 내용 있으면 DB 자체 장애 의심
- dto 전체가 찍혀있어서 데이터 자체로 추적
- prismaErrorCode / error 포함으로 원인 파악 속도 개선

handle-failed.log
- handle() 실패 시 기록 (GatherWorker)
- job_queue에는 있는 것 (jobId 있음)
- 내용 있으면 외부 API(보험사) 장애 의심
- jobId로 job_queue SELECT해서 상세 파악 가능
- 퇴근 전 이 파일 확인 후 job_queue 수동 처리
```
